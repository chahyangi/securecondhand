# securecondhand 개발 과정

이 문서는 `중고거래플랫폼_설계문서.md`(요구사항/설계)와 별개로, **실제로 코드가 어떤 순서와 판단으로 만들어졌는지**를 정리한 기록이다. git 커밋 이력과 설계문서 7장(구현 현황)을 바탕으로 작성했다.

> 과제 원본은 `secure-coding-slide.v2.pdf`(저장소 루트). 최소 요구사항 중 **유저 간 송금**은 2.5절에서 내부 포인트/잔액 기반으로 구현을 완료했다(`과제명세.md`, 설계문서 7.1/7.2절).

## 1. 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| 프론트엔드 | React + Vite | 모바일 웹 전용, 최대 폭 480px 고정 레이아웃 |
| 백엔드 | Django + Django REST Framework | REST API |

### 2.1 요구사항 → 설계

### 2.2 1차 구현 (Initial commit, 2026-07-21)
설계문서를 바탕으로 프론트/백엔드를 한 번에 구축한 첫 커밋. 50개 파일, 약 7,250줄.

- **백엔드**: `models.py`(profiles/friend_requests/products/product_images/wishlists/chatrooms/chat_participants/chat_messages/trade_verifications/verification_steps/notification_settings/reports), `views.py`(538줄, REST 엔드포인트 전체), `consumers.py`(WebSocket 채팅), `ws_auth.py`(쿼리스트링 토큰 인증 미들웨어), `serializers.py`, `urls.py`, 마이그레이션 2개.
- **프론트엔드**: `App.jsx`(라우팅), `AuthContext.jsx`(로그인 상태 관리), 화면 4종(`MainGrid`/`ProductDetail`/`Chat`/`Settings`) + `Login`/`Signup`/`ProductCreate`/`AdminDashboard`, 공용 `api.js`(365줄, 전체 API 클라이언트), `mock/data.js`(백엔드 미기동 시 폴백용 더미 데이터).
- **인프라**: `docker-compose.yml`(Postgres), `start.sh`/`npm start`(DB→백엔드→프론트 순차 기동 스크립트).

이 시점에 회원가입/로그인, 상품 CRUD+이미지 업로드, 무한 스크롤 그리드, 찜, 신고, 친구 관리, 실시간 채팅(대리인 초대/자동 퇴장 포함), 대면 거래 인증 체인, 알림 설정, 관리자 대시보드까지 기능적으로는 대부분 동작하는 상태로 완성됐다.

### 2.3 2차 개선 (163b1cf, 2026-07-21 22:30)
1차 구현을 실제로 써보면서 발견한 3가지 구멍을 메운 커밋.

1. **채팅방 목록 화면 신설** (`src/pages/ChatList.jsx`, 신규 97줄) — 이전에는 하단 네비 "채팅" 탭이 특정 상품의 채팅방으로 바로 들어가는 구조였는데, 내가 참여 중인 모든 채팅방을 상대방/상품/거래상태와 함께 보여주는 `/chats` 목록 화면으로 교체.
2. **찜 목록 화면 실제 연결** (`Settings.jsx`) — 설정 › 찜 목록 메뉴가 눌러도 메인화면으로 리다이렉트만 되던 것을, 실제로 찜한 상품을 조회/해제할 수 있도록 연결.
3. **대면 거래 인증 위조 방지 강화** (`backend/marketplace/views.py`) — 클라이언트가 보내는 `from_user`/`to_user` 값을 그대로 신뢰하던 로직을, 서버가 채팅방 참여자 목록에서 직접 판매자→대리인→구매자 체인을 계산해 "지금 눌러야 할 사람"과 실제 요청자가 다르면 403으로 거부하도록 수정. (설계문서 7.2절의 "대면 거래 인증 위조 방지" 항목)
4. **토큰 저장소 변경** (`AuthContext` 관련) — `localStorage`는 브라우저의 모든 탭이 공유해 여러 계정으로 동시 테스트할 때 토큰이 서로 덮어써지는 문제가 있어, 탭/창마다 독립적인 `sessionStorage`로 변경.

### 2.5 송금 기능 구현 (이번 세션, 2026-07-22)
과제 최소 요구사항(`secure-coding-slide.v2.pdf` 35p, `과제명세.md`)에 있는 "유저 간 송금"이 그동안 미구현으로 남아 있었다. `products`에 가격 필드조차 없어 실제 결제 PG 연동 없이는 구현이 애매한 상태였는데, 마감까지 하루 이상 여유가 있는 것으로 확인되어(제출 기한이 24일(금)로 정정됨) 이번 세션에서 범위를 정해 구현했다.

- **범위 결정**: 실제 PG 계약/키 발급은 하루 안에 불가능하므로, 실물 결제 대신 **내부 포인트/잔액 시스템**으로 "유저 간 송금"을 구현하기로 결정 (에스크로는 범위 밖으로 유지).
- **모델 추가**: `Product.price`(원 단위), `Profile.balance`(가입 시 기본 1,000,000 지급), `Transfer`(sender/receiver/amount/memo/chatroom, 불변 원장) — `backend/marketplace/models.py`.
- **동시성/이중지불 방지**: `POST /transfers`가 sender/receiver 두 `Profile` 행을 `user_id` 오름차순으로 `select_for_update()`한 뒤 같은 트랜잭션 안에서 잔액 검증 → 차감/입금 → `Transfer` 생성을 원자적으로 처리. 잠금 순서를 항상 고정해 두 사람이 동시에 서로에게 송금해도 교착상태에 빠지지 않도록 함. curl로 동시 요청 10개를 보내 잔액이 정확히 소진되고 절대 음수로 내려가지 않음을 확인.
- **위조/오남용 방지**: `sender`는 항상 `request.user`로 서버가 강제(클라이언트가 지정 불가), 본인에게 송금·0원 이하 송금은 DB 체크 제약과 서버 검증 양쪽에서 차단, `balance`는 `ProfileSerializer`에서 읽기 전용이라 `PATCH /profiles/me`로 직접 조작 불가.
- **프론트**: 설정 › 송금 화면(잔액 조회, 아이디로 송금, 내역 조회)과 채팅방 안 "송금하기"(구매자→판매자, 상품 가격을 기본값으로 제안) 두 진입점을 만듦 (`src/pages/Settings.jsx`의 `WalletView`, `src/pages/Chat.jsx`).
- **검증**: `python manage.py check`/`migrate` 통과, curl로 정상 송금·자기송금 차단·마이너스/0원 차단·잔액부족 차단·원장 불변(PATCH/DELETE 405)·동시 요청 레이스 컨디션까지 확인. `npm run build` 통과.

### 2.6 악성 유저·상품 자동 차단 구현 (이번 세션, 2026-07-22)
사용자가 "악성 유저/상품 차단이 실제로 있냐"고 물어봐서 코드를 확인해보니, 신고 접수(`Report`)와 관리자 수동 처리(사용자 정지/상품 삭제/신고 처리완료)만 있고, 설계문서 25p에 명시했던 **"일정 횟수 이상 신고되면 자동으로 차단/휴면 전환"** 로직 자체가 빠져 있었다. 이 항목은 과제 Group A(필수 포함)에 속해서 구현했다.

- **임계값 기준을 "신고 횟수"가 아니라 "서로 다른 신고자 수"로 설계**: 신고 횟수 그대로 세면 악성 유저 한 명이 같은 대상을 반복 신고해서 혼자 임계값을 채워 버릴 수 있다(신고 기능 자체를 무기화). `Report.objects.filter(target_type=, target_id=).values('reporter').distinct().count()`로 세서, 서로 다른 신고자 3명 이상이 모여야만 제재가 걸리도록 함 (`views.py`의 `apply_auto_moderation`).
- **제재 방식**: 상품은 `Product.is_blocked=True`로 표시되어 `ProductViewSet.get_queryset`에서 스태프가 아닌 모든 요청(소유자 포함)에서 제외됨(목록/상세 모두 404). 유저는 `is_active=False`로 전환되어 기존 관리자 수동 정지와 동일하게 로그인이 즉시 막힘. 관리자 화면(`AdminProductViewSet`/`AdminUserViewSet`)은 별도 큐어리셋이라 차단된 항목도 그대로 보여 검토·해제가 가능하다.
- **검증**: curl로 (1) 서로 다른 신고자 3명이 신고 → 상품이 200에서 404로 전환됨, (2) 같은 신고자 1명이 5번 반복 신고 → 차단 안 됨(어뷰징 방지 확인), (3) 서로 다른 신고자 3명이 유저를 신고 → 그 유저 로그인이 400으로 막힘, (4) 관리자 계정으로는 차단된 상품을 여전히 조회 가능함을 확인.

### 2.7 토스페이먼츠 테스트 연동 — 지갑 충전 (이번 세션, 2026-07-22)
2.5절의 내부 잔액 송금이 "화면상으로는 우리 앱 자체 폼일 뿐"이라는 한계가 있어서, 사용자가 진짜 PG 결제 흐름을 화면으로 보고 싶어 해 토스페이먼츠 **테스트 연동**(사업자 등록 불필요, 개발자 가입만으로 발급되는 테스트 키)을 추가했다.

- **범위**: PG 결제는 항상 가맹점(우리 서비스) 정산 계좌로 들어오지 특정 개인에게 바로 가지 않는다. 그래서 "유저 A → 유저 B 실제 송금"까지는 대체할 수 없고, 실제로 붙일 수 있는 건 **"내 지갑 충전"**뿐이다 — 충전 후 유저 간 이동은 여전히 2.5절의 내부 잔액 방식 그대로. 이 한계를 사용자에게 먼저 설명하고 진행에 동의를 받았다.
- **키 확인**: 발급받은 `TOSS_CLIENT_KEY`가 `test_ck_...`(API 개별연동/결제창 키)였다 — 처음에 `결제위젯`(gck/gsk) 방식으로 설계했다가, 실제 키 프리픽스를 curl로 확인(`GET /api/payments/config/`)한 뒤 `tossPayments.payment({customerKey}).requestPayment({method:'CARD', ...})` 방식(API 개별연동/결제창)으로 다시 구현했다. 두 방식은 키 종류가 다르면 `INVALID_API_KEY` 에러가 나므로 반드시 맞춰야 한다.
- **모델**: `PaymentOrder`(order_id, user, amount, status: pending/confirmed/failed, payment_key) — 결제창을 열기 전에 서버가 먼저 금액을 기록해두는 용도.
- **API**: `GET /payments/config`(공개 client key 제공), `POST /payments/orders`(주문 생성), `POST /payments/orders/confirm`(토스 결제 승인 API를 시크릿 키로 서버 간 호출 후 잔액 충전).
- **위조/중복 방지**: 승인 요청의 금액이 주문 생성 시 기록해둔 금액과 다르면 토스에 확인하기 전에 즉시 차단. 이미 승인 완료된 주문을 다시 confirm 요청해도 잔액을 중복 적립하지 않음(멱등성).
- **검증**: curl로 (1) 주문 생성 정상 동작, (2) 승인 요청 금액을 주문 금액과 다르게 보내면 토스 호출 전에 400으로 차단, (3) 가짜 `payment_key`로 승인 요청 → 실제 토스 서버가 "결제 진행 데이터 없음" 오류를 반환하고 주문이 `failed`로 전환됨(시크릿 키가 유효하고 실제로 토스 서버와 통신되는 것까지 확인), (4) DB에서 주문을 `confirmed`로 만든 뒤 같은 주문을 다시 confirm 요청해도 잔액이 중복 적립되지 않음을 확인. `npm run build` 통과.
- **미검증**: 브라우저에서 실제로 결제창을 열고 토스 문서의 테스트 카드번호로 결제를 완료하는 **성공 경로 전체(화면 클릭 → 리다이렉트 → 승인 → 잔액 반영)**는 이 세션에서 브라우저 환경이 없어 직접 확인하지 못했다. 사용자가 직접 `/wallet/topup`에서 시도해봐야 한다.

### 2.8 `off.sh` 추가 (이번 세션, 2026-07-22)
`start.sh`(`npm start`)는 `Ctrl + C`로 백엔드·프론트엔드를 종료하지만 Postgres 컨테이너는 의도적으로 계속 띄워둔다. 그런데 이번 세션에서 curl 테스트용으로 백그라운드에 띄운 `manage.py runserver`를 정리하다가, 터미널을 그냥 닫아버리면(트랩이 안 걸리는 경우) 프로세스가 남아 포트를 계속 점유하는 걸 직접 겪었다. 이를 한 번에 정리하는 `off.sh`(`npm stop`)를 추가했다 — 백엔드/프론트엔드 프로세스 종료 + Postgres 컨테이너 정지(`docker compose stop`, 데이터 보존)를 한 번에 처리하며, 이미 다 꺼져 있어도 에러 없이 끝난다(포트 점유/컨테이너 상태를 실제로 띄웠다 꺼보며 확인).

## 3. 구현 중 발견/해결한 주요 이슈

설계 단계에는 없었지만 구현하면서 드러나 수정한 것들 (설계문서 7.2절 기준).

| 이슈 | 해결 |
|---|---|
| 상품 이미지 업로드/상태 변경에 소유권 검증 누락 | 로그인만 하면 남의 상품도 건드릴 수 있던 취약점 발견 → `IsOwnerOrReadOnly` 권한 클래스 추가 |
| 대면 거래 인증 위조 가능 | 클라이언트가 보낸 `from_user`/`to_user`를 그대로 신뢰 → 서버가 참여자 목록에서 체인을 직접 계산하도록 변경 (2.3절 3번) |
| 사진 순서 변경 시 `IntegrityError` | `(product_id, order)` 유니크 제약 때문에 두 사진 순서를 바로 맞바꾸면 충돌 → 임시로 1만 이상 값으로 옮긴 뒤 최종 순서를 저장하는 2단계 갱신으로 처리 |
| WebSocket이 커스텀 인증 헤더를 못 보냄 | 토큰을 쿼리스트링(`?token=`)으로 전달받아 인증하는 미들웨어(`ws_auth.py`)를 별도 구현 |
| 가격 체계 미설계 | 송금/에스크로 요구사항이 있었지만 `products`에 price 컬럼이 없어, 결제 PG 연동 없이는 의미 있는 구현이 어렵다고 판단해 이번 범위에서 보류 |

---

## 4. 현재 상태 요약

- **완료**: 회원/인증, 상품 CRUD+이미지+가격, 무한 스크롤 그리드, 찜, 신고(서로 다른 신고자 3명 이상 시 자동 차단/휴면 전환 포함), 친구 관리, 실시간 채팅(대리인 포함), 대면 거래 인증 체인, 유저 간 송금(내부 잔액) + 토스페이먼츠 테스트 연동 지갑 충전, 알림 설정 저장, 관리자 대시보드.
- **미구현(설계로만 존재)**: 소셜 로그인(OAuth), 에스크로(보류결제), 실제 푸시 발송(FCM/APNs), 판매자 측 대리인 초대 UI.
- **개발 환경 한계**: WebSocket 채널 레이어가 `InMemoryChannelLayer`라 단일 프로세스에서만 동작 (스케일 아웃하려면 Redis 기반 `channels_redis` 필요). 이미지도 로컬 디스크 저장이라 실서비스 전환 시 오브젝트 스토리지(S3 등)로 교체 필요.

더 자세한 내용(화면/DB/API 설계 전체, 실행 방법)은 `중고거래플랫폼_설계문서.md`를 참고.
