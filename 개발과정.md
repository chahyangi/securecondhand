# securecondhand 개발 과정

이 문서는 `중고거래플랫폼_설계문서.md`(요구사항/설계)와 별개로, **실제로 코드가 어떤 순서와 판단으로 만들어졌는지**를 정리한 기록이다. git 커밋 이력과 설계문서 7장(구현 현황)을 바탕으로 작성했다.

> 과제 원본은 `secure-coding-slide.v2.pdf`(저장소 루트). 최소 요구사항 중 **유저 간 송금**은 2.5절에서 내부 포인트/잔액 기반으로 구현을 완료했다(`과제명세.md`, 설계문서 7.1/7.2절).

---

## 1. 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| 프론트엔드 | React + Vite | 모바일 웹 전용, 최대 폭 480px 고정 레이아웃 |
| 백엔드 | Django + Django REST Framework | REST API |
| 실시간 통신 | Django Channels + Daphne | WebSocket 채팅/알림, `runserver` 한 프로세스로 REST와 함께 서빙 |

| 인증 | DRF `TokenAuthentication` | 세션 쿠키 대신 토큰 방식 채택 (아래 4장 참고) |

설계문서를 바탕으로 프론트/백엔드를 한 번에 구축한 첫 커밋. 50개 파일, 약 7,250줄.

이 시점에 회원가입/로그인, 상품 CRUD+이미지 업로드, 무한 스크롤 그리드, 찜, 신고, 친구 관리, 실시간 채팅(대리인 초대/자동 퇴장 포함), 대면 거래 인증 체인, 알림 설정, 관리자 대시보드까지 기능적으로는 대부분 동작하는 상태로 완성됐다.

### 2.3 2차 개선 (163b1cf, 2026-07-21 22:30)

4. **토큰 저장소 변경** (`AuthContext` 관련) — `localStorage`는 브라우저의 모든 탭이 공유해 여러 계정으로 동시 테스트할 때 토큰이 서로 덮어써지는 문제가 있어, 탭/창마다 독립적인 `sessionStorage`로 변경.

### 2.5 송금 기능 구현 (이번 세션, 2026-07-22)
과제 최소 요구사항(`secure-coding-slide.v2.pdf` 35p, `과제명세.md`)에 있는 "유저 간 송금"이 그동안 미구현으로 남아 있었다. `products`에 가격 필드조차 없어 실제 결제 PG 연동 없이는 구현이 애매한 상태였는데, 마감까지 하루 이상 여유가 있는 것으로 확인되어(제출 기한이 24일(금)로 정정됨) 이번 세션에서 범위를 정해 구현했다.

- **범위 결정**: 실제 PG 계약/키 발급은 하루 안에 불가능하므로, 실물 결제 대신 **내부 포인트/잔액 시스템**으로 "유저 간 송금"을 구현하기로 결정 (에스크로는 범위 밖으로 유지).
- **모델 추가**: `Product.price`(원 단위), `Profile.balance`(가입 시 기본 1,000,000 지급), `Transfer`(sender/receiver/amount/memo/chatroom, 불변 원장) — `backend/marketplace/models.py`.
- **동시성/이중지불 방지**: `POST /transfers`가 sender/receiver 두 `Profile` 행을 `user_id` 오름차순으로 `select_for_update()`한 뒤 같은 트랜잭션 안에서 잔액 검증 → 차감/입금 → `Transfer` 생성을 원자적으로 처리. 잠금 순서를 항상 고정해 두 사람이 동시에 서로에게 송금해도 교착상태에 빠지지 않도록 함. curl로 동시 요청 10개를 보내 잔액이 정확히 소진되고 절대 음수로 내려가지 않음을 확인.
- **위조/오남용 방지**: `sender`는 항상 `request.user`로 서버가 강제(클라이언트가 지정 불가), 본인에게 송금·0원 이하 송금은 DB 체크 제약과 서버 검증 양쪽에서 차단, `balance`는 `ProfileSerializer`에서 읽기 전용이라 `PATCH /profiles/me`로 직접 조작 불가.
- **프론트**: 설정 › 송금 화면(잔액 조회, 아이디로 송금, 내역 조회)과 채팅방 안 "송금하기"(구매자→판매자, 상품 가격을 기본값으로 제안) 두 진입점을 만듦 (`src/pages/Settings.jsx`의 `WalletView`, `src/pages/Chat.jsx`).
- **검증**: `python manage.py check`/`migrate` 통과, curl로 정상 송금·자기송금 차단·마이너스/0원 차단·잔액부족 차단·원장 불변(PATCH/DELETE 405)·동시 요청 레이스 컨디션까지 확인. `npm run build` 통과.

### 2.6 악성 유저·상품 자동 차단 구현 (이번 세션, 2026-07-22)
사용자가 "악성 유저/상품 차단이 실제로 있냐"고 물어봐서 코드를 확인해보니, 신고 접수(`Report`)와 관리자 수동 처리(사용자 정지/상품 삭제/신고 처리완료)만 있고, 설계문서 25p에 명시했던 **"일정 횟수 이상 신고되면 자동으로 차단/휴면 전환"** 로직 자체가 빠져 있었다. 이 항목은 과제 Group A(필수 포함)에 속해서 구현했다.

- **임계값 기준을 "신고 횟수"가 아니라 "서로 다른 신고자 수"로 설계**: 신고 횟수 그대로 세면 악성 유저 한 명이 같은 대상을 반복 신고해서 혼자 임계값을 채워 버릴 수 있다(신고 기능 자체를 무기화). `Report.objects.filter(target_type=, target_id=).values('reporter').distinct().count()`로 세서, 서로 다른 신고자 3명 이상이 모여야만 제재가 걸리도록 함 (`views.py`의 `apply_auto_moderation`).
- **제재 방식**: 상품은 `Product.is_blocked=True`로 표시되어 `ProductViewSet.get_queryset`에서 스태프가 아닌 모든 요청(소유자 포함)에서 제외됨(목록/상세 모두 404). 유저는 `is_active=False`로 전환되어 기존 관리자 수동 정지와 동일하게 로그인이 즉시 막힘. 관리자 화면(`AdminProductViewSet`/`AdminUserViewSet`)은 별도 큐어리셋이라 차단된 항목도 그대로 보여 검토·해제가 가능하다.
- **검증**: curl로 (1) 서로 다른 신고자 3명이 신고 → 상품이 200에서 404로 전환됨, (2) 같은 신고자 1명이 5번 반복 신고 → 차단 안 됨(어뷰징 방지 확인), (3) 서로 다른 신고자 3명이 유저를 신고 → 그 유저 로그인이 400으로 막힘, (4) 관리자 계정으로는 차단된 상품을 여전히 조회 가능함을 확인.

### 2.7 토스페이먼츠 테스트 연동 — 지갑 충전 (이번 세션, 2026-07-22)
2.5절의 내부 잔액 송금이 "화면상으로는 우리 앱 자체 폼일 뿐"이라는 한계가 있어서, 사용자가 진짜 PG 결제 흐름을 화면으로 보고 싶어 해 토스페이먼츠 **테스트 연동**(사업자 등록 불필요, 개발자 가입만으로 발급되는 테스트 키)을 추가했다.

- **범위**: PG 결제는 항상 가맹점(우리 서비스) 정산 계좌로 들어오지 특정 개인에게 바로 가지 않는다. 그래서 "유저 A → 유저 B 실제 송금"까지는 대체할 수 없고, 실제로 붙일 수 있는 건 **"내 지갑 충전"**뿐이다 — 충전 후 유저 간 이동은 여전히 2.5절의 내부 잔액 방식 그대로. 이 한계를 사용자에게 먼저 설명하고 진행에 동의를 받았다.
- **키 확인**: 발급받은 `TOSS_CLIENT_KEY`가 `test_ck_...`(API 개별연동/결제창 키)였다 — 처음에 `결제위젯`(gck/gsk) 방식으로 설계했다가, 실제 키 프리픽스를 curl로 확인(`GET /api/payments/config/`)한 뒤 `tossPayments.payment({customerKey}).requestPayment({method:'CARD', ...})` 방식(API 개별연동/결제창)으로 다시 구현했다. 두 방식은 키 종류가 다르면 `INVALID_API_KEY` 에러가 나므로 반드시 맞춰야 한다.
- **모델**: `PaymentOrder`(order_id, user, amount, status: pending/confirmed/failed, payment_key) — 결제창을 열기 전에 서버가 먼저 금액을 기록해두는 용도.
- **API**: `GET /payments/config`(공개 client key 제공), `POST /payments/orders`(주문 생성), `POST /payments/orders/confirm`(토스 결제 승인 API를 시크릿 키로 서버 간 호출 후 잔액 충전).
- **위조/중복 방지**: 승인 요청의 금액이 주문 생성 시 기록해둔 금액과 다르면 토스에 확인하기 전에 즉시 차단. 이미 승인 완료된 주문을 다시 confirm 요청해도 잔액을 중복 적립하지 않음(멱등성).
- **검증**: curl로 (1) 주문 생성 정상 동작, (2) 승인 요청 금액을 주문 금액과 다르게 보내면 토스 호출 전에 400으로 차단, (3) 가짜 `payment_key`로 승인 요청 → 실제 토스 서버가 "결제 진행 데이터 없음" 오류를 반환하고 주문이 `failed`로 전환됨(시크릿 키가 유효하고 실제로 토스 서버와 통신되는 것까지 확인), (4) DB에서 주문을 `confirmed`로 만든 뒤 같은 주문을 다시 confirm 요청해도 잔액이 중복 적립되지 않음을 확인. `npm run build` 통과.
- **미검증**: 브라우저에서 실제로 결제창을 열고 토스 문서의 테스트 카드번호로 결제를 완료하는 **성공 경로 전체(화면 클릭 → 리다이렉트 → 승인 → 잔액 반영)**는 이 세션에서 브라우저 환경이 없어 직접 확인하지 못했다. 사용자가 직접 `/wallet/topup`에서 시도해봐야 한다.

### 2.8 `off.sh` 추가 (이번 세션, 2026-07-22)
`start.sh`(`npm start`)는 `Ctrl + C`로 백엔드·프론트엔드를 종료하지만 Postgres 컨테이너는 의도적으로 계속 띄워둔다. 그런데 이번 세션에서 curl 테스트용으로 백그라운드에 띄운 `manage.py runserver`를 정리하다가, 터미널을 그냥 닫아버리면(트랩이 안 걸리는 경우) 프로세스가 남아 포트를 계속 점유하는 걸 직접 겪었다. 이를 한 번에 정리하는 `off.sh`(`npm stop`)를 추가했다 — 백엔드/프론트엔드 프로세스 종료 + Postgres 컨테이너 정지(`docker compose stop`, 데이터 보존)를 한 번에 처리하며, 이미 다 꺼져 있어도 에러 없이 끝난다(포트 점유/컨테이너 상태를 실제로 띄웠다 꺼보며 확인).

### 2.9 로고 교체 + 관리자 신고 처리에 실제 제재 연결 (2026-07-23)
- **로고**: 임시 워드마크 `마당`을 프로젝트명에 맞춘 `secure`/`cond`/`hand` 3줄 로고 + 자물쇠 아이콘으로 교체했다(`src/components/Common.jsx`의 `Logo`). 자물쇠 열쇠 구멍은 흔한 원+세모 모양 대신 사각형(box)으로 SVG `mask`를 써서 뚫었다. 브라우저가 없는 환경이라 Playwright로 실제 렌더링 스크린샷을 뽑아 확인했다.
- **관리자 신고 처리 — "처리완료"에 후속 조치가 없던 문제**: `AdminReportViewSet.resolve`가 신고의 `status`만 `resolved`로 바꿀 뿐, 신고 대상(상품/유저)에는 아무 조치도 하지 않는다는 걸 사용자가 지적해서 확인했다. 신고 3건(서로 다른 신고자) 자동 차단(2.6절)과는 별개로, 관리자가 신고 1건만 보고도 직접 판단해서 즉시 제재할 수 있는 `sanction` 액션(`PATCH /admin/reports/{id}/sanction/`)을 추가했다 — 상품이면 `is_blocked=True`, 유저면 `is_active=False`로 만들면서 신고도 함께 처리완료 처리한다. 기존 "처리완료"는 "검토했지만 조치 불필요"용으로 그대로 남겨뒀다. curl로 신고 1건만으로 상품이 차단되는지, 일반 계정은 403으로 막히는지 확인했다.
- **유저 신고 UI가 아예 없던 문제**: 위 기능을 만들고 나서 사용자가 "유저 차단 없는데??"라고 재차 지적해서 코드를 다시 훑어보니, `createReport`를 호출하는 곳이 `ProductDetail.jsx`의 상품 신고 버튼 하나뿐이었고 `target_type: 'product'`로 고정돼 있었다 — 즉 유저를 신고할 UI 경로 자체가 없어서, 신고 3건 자동 휴면·관리자 유저 정지 기능 모두 curl로만 트리거 가능하고 실제 사용 흐름에서는 닿을 수 없는 죽은 코드였다. 상품 상세의 "신고" 버튼을 모달로 바꿔 "이 상품"/"판매자" 중 신고 대상을 고를 수 있게 했다(`ProductDetail.jsx`). 본인 상품에는 애초에 신고 버튼 자체가 안 보이므로 셀프 신고는 구조적으로 불가능하다.
- **차단 해제가 없던 문제**: 사용자가 "차단을 해제할 수는 없나?"라고 물어봐서 확인해보니, 유저는 `is_active` 토글로 정지/정지해제가 이미 양방향으로 되는데 상품은 `is_blocked`를 다시 되돌릴 방법이 관리자 화면에 없었다(강제 삭제만 있었음). `AdminProductViewSet`에 `block` 액션(`PATCH /admin/products/{id}/block/`, `is_blocked` 토글)을 추가해 자동 차단이든 관리자 sanction이든 관계없이 오탐 시 되돌릴 수 있게 했다. curl로 차단→404, 해제→200, 일반 계정 403을 확인했다.
- **처리완료해도 신고 1건만 더 들어오면 즉시 재차단되던 버그**: 사용자가 "처리 완료 했는데 한번 더 신고자가 신고 들어가면 다시 차단되네?"라고 지적해서 원인을 찾았다. `apply_auto_moderation`이 신고 상태(대기중/처리완료)를 구분하지 않고 **모든** 신고를 서로 다른 신고자 수로 세고 있었다 — 그래서 신고 3건으로 차단된 뒤 관리자가 셋 다 처리완료하고 차단을 풀어도, 새 신고가 딱 1건만 더 들어오면 "처리완료된 옛날 신고 + 새 신고 1건"이 다시 3건으로 합산돼 즉시 재차단됐다. `Report.Status.PENDING`인 신고만 세도록 고쳤고, 관리자가 "차단 해제"/"정지 해제"를 누를 때도 관련 대기중 신고를 함께 처리완료 처리하도록 해서 어느 경로로 풀어도 같은 문제가 재발하지 않게 했다(`AdminProductViewSet.block`, `AdminUserViewSet.suspend`). curl로 (1) 신고 3건 → 차단 → 전부 처리완료 → 차단 해제 → 신고 1건 추가 → **차단 안 됨**(수정 확인), (2) 대기중 신고가 그대로 남은 상태에서 신고가 새로 쌓여 3건이 되면 여전히 정상적으로 재차단됨을 확인했다.
- **채팅 사진이 깨져 보이던 버그**: 사용자가 "사진 깨지는데?"라고 지적해서 원인을 찾았다. `ChatRoomViewSet.upload_image`가 `default_storage.url(path)`를 그대로 반환하고 있었는데, 이 값은 스킴/호스트 없는 절대경로(`/media/chat/...`)일 뿐이다. 상품 사진은 `ProductImageSerializer`의 DRF `ImageField`가 `request.build_absolute_uri()`로 자동으로 감싸줘서 `http://localhost:8000/media/...` 같은 완전한 URL이 되는데, 채팅 사진 업로드는 시리얼라이저를 안 쓰고 직접 응답을 만들다 보니 이 처리가 빠져 있었다. 그 결과 프론트(vite, 5173)가 `/media/...`를 자기 자신의 origin으로 잘못 해석했고 — vite 프록시 설정(`vite.config.js`)엔 `/api`, `/ws`만 있고 `/media`는 없어서 — 채팅 이미지 요청이 404/SPA 폴백으로 떨어져 깨진 이미지로 보였다. `upload_image`도 `request.build_absolute_uri()`로 감싸도록 고쳐서 상품 사진과 동일한 절대 URL을 반환하게 했다. curl로 업로드 응답이 `http://localhost:8000/media/chat/...` 형태로 바뀐 것과, 그 URL을 직접 요청했을 때 실제 PNG 바이트(200, `image/png`)가 오는 것까지 확인했다.

### 2.10 대면 거래 인증 완료 시 자동 송금 (2026-07-23)
사용자가 "인수인증 인계인증 구매인증 판매인증 완료되면 각자 계좌로 돈 전송되게 만들어놔"라고 요청해서, 대면 거래 인증 체인이 끝까지 완료되는 순간 구매자→판매자로 상품 가격만큼 자동으로 송금되도록 구현했다.

- **공통 로직 분리**: 기존 `TransferViewSet.create`에 있던 "두 `Profile.balance`를 `select_for_update`로 잠그고 원자적으로 이동 + `Transfer` 생성" 로직을 `execute_transfer(sender, receiver, amount, memo, chatroom)` 함수로 분리했다(잔액 부족 시 `InsufficientBalanceError`). `TransferViewSet.create`도 이 함수를 쓰도록 리팩터링해서 로직이 두 곳에 중복되지 않게 했다.
- **트리거 지점**: `ChatRoomViewSet.verification`에서 한 구간(hop)이 give+receive 모두 완료되고, 그 구간이 **체인의 마지막 구간**일 때 자동 송금을 실행한다. "마지막 구간인지"는 클라이언트가 보낸 `is_last_hop`을 믿지 않고 `hop_index == len(hops) - 1`로 서버가 직접 계산한다 — 결제가 걸린 로직이라 클라이언트 입력을 신뢰하면 조작 위험이 있어서다.
- **송금 대상/금액**: 체인에서 role이 `buyer`/`seller`인 참여자를 찾아 구매자→판매자로 `product.price`만큼 송금. 대리인은 거래 당사자가 아니므로 금액 이동에 관여하지 않는다.
- **이중 결제 방지**: 자동 송금 실행 전에 `Transfer.objects.filter(chatroom=chatroom).exists()`로 이미 이 채팅방에서 송금(수동 "송금하기" 포함)이 있었는지 확인하고, 있으면 건너뛴다.
- **잔액 부족 시**: 거래 확정 자체(대면 인증)는 그대로 진행시키되(물리적 인수인계는 이미 일어난 사실이므로), 자동 송금만 실패시키고 채팅방에 "구매자 잔액 부족으로 자동 송금 실패, 설정 › 송금에서 직접 송금해달라"는 시스템 메시지를 남긴다.
- **검증(curl)**: (1) 정상 잔액으로 인증 완료 → 구매자 -가격, 판매자 +가격, 시스템 메시지에 자동 송금 안내 확인. (2) 구매자 잔액 0으로 인증 완료 → 거래는 confirmed로 바뀌지만 잔액은 그대로, "잔액 부족" 시스템 메시지 확인. (3) 인증 완료 전에 구매자가 같은 채팅방에서 수동으로 이미 송금해둔 경우 → 인증 완료 시 자동 송금이 건너뛰어져 이중 차감이 안 됨을 확인(해당 채팅방의 `Transfer` 레코드가 정확히 1건).
- 이번 세션에서 테스트/개발 편의를 위해 **모든 유저의 잔액을 0으로 초기화**했다(`Profile.objects.all().update(balance=0)`) — 이제 지갑 충전(토스페이먼츠 테스트 결제, 2.7절)을 실제로 거쳐야 잔액이 생기고, 그 잔액으로 거래를 완료하는 전체 흐름을 처음부터 시연할 수 있다.

### 2.11 결제 성공해도 충전 안 되던 버그 — HashRouter 리다이렉트 불일치 (2026-07-23)
사용자가 브라우저에서 실제로 토스 테스트 결제를 성공시켰는데도 잔액이 충전되지 않는다고 지적해서 찾았다. 이 앱은 `main.jsx`에서 **`HashRouter`**를 쓰는데(`/#/wallet/topup` 형태), 2.7절에서 만든 `TopUp.jsx`의 `successUrl`/`failUrl`은 `#` 없이 `${origin}/wallet/topup/result`로 넘기고 있었다.

토스는 결제 완료 후 이 URL 뒤에 `paymentKey`/`orderId`/`amount`(실패 시 `code`/`message`) 쿼리스트링을 그대로 붙여서 리다이렉트시키는데, `#`가 없으면 HashRouter는 이 요청을 "해시 뒤에 아무 경로도 없음" = 루트(`/`)로 인식해서 `TopUpResult` 대신 메인 화면(`MainGrid`)을 그대로 렌더링했다. 그래서 겉보기엔 결제가 끝나고 메인 화면으로 튕겨온 것처럼만 보이고, `confirmPaymentOrder`(잔액을 실제로 충전하는 API 호출)는 아예 실행조차 되지 않았다 — 결제는 토스 쪽에서 성공 처리됐지만 우리 서버는 그 사실을 전혀 통보받지 못한 셈이다.

`successUrl`/`failUrl`을 `${origin}/#/wallet/topup/result`로 수정해서, 토스가 그 뒤에 쿼리스트링을 붙여도 `#/wallet/topup/result?paymentKey=...` 형태가 되어 HashRouter가 `TopUpResult` 라우트로 정확히 매칭하도록 고쳤다. Playwright로 실제 브라우저에서 이 형태의 URL을 직접 열어, 화면에 "충전 결과"(`TopUpResult`) 타이틀이 뜨고 `confirmPaymentOrder` 호출이 실제로 나가는 것(인증 안 된 상태라 401 에러가 응답으로 옴)까지 확인했다 — 로그인된 정상 세션이라면 이 호출이 성공해 잔액이 충전된다.

**후속 버그**: `#`를 붙였는데도 사용자가 재시도 후 "결제 정보가 올바르지 않아요"를 봤다. 원인은 토스가 `successUrl`을 문자열로 그대로 이어붙이는 게 아니라 `URL` 객체로 파싱해서 쿼리스트링을 붙이는 것으로 보인다는 점이었다 — URL 스펙상 쿼리스트링은 `#`(fragment) **앞**에 와야 하므로, 실제 리다이렉트는 `http://host/?paymentKey=...#/wallet/topup/result` 형태로 왔다. `useSearchParams()`(HashRouter 버전)는 `#` **안쪽** 쿼리스트링만 읽기 때문에, `#` 앞에 있는 이 값들을 전혀 못 봤다 — 그래서 "테스트 API라 막히는 건가?"처럼 보였지만 실제로는 순수 프론트 라우팅 파싱 문제였다. (사용자가 이 시점에 "일정 시간 지나면 그냥 통과시키는 우회 로직"을 제안했는데, 그건 결제 확인 절차를 완전히 무력화하는 심각한 보안 구멍이라 채택하지 않고 근본 원인을 마저 찾아 고쳤다.) `TopUpResult.jsx`가 `useSearchParams()` 대신(또는 함께) `new URLSearchParams(window.location.search)`로 `#` 앞의 진짜 쿼리스트링도 읽도록 수정. Playwright로 쿼리스트링을 `#` 앞/뒤 두 가지 형태 모두 재현해 두 경우 다 `confirmPaymentOrder`가 정상 호출됨을 확인했다.

### 2.12 거래 완료 시 상품이 자동으로 "판매완료"로 안 바뀌던 문제 (2026-07-23)
사용자가 "구매인증 + 판매인증 했는데 판매완료 자동으로 안 되는 거 같은데?"라고 지적했다. 확인해보니 `ChatRoom.status`(협의중→인계완료/거래확정→거래완료)와 `Product.status`(판매중/예약중/판매완료)는 완전히 별개 필드였고, `Product.status`는 판매자가 상품 상세에서 상태 칩을 수동으로 눌러야만 바뀌었다 — 대면 거래 인증이 끝까지 완료돼도 상품 자체는 계속 "판매중"으로 남아 있었다.

2.10절의 자동 송금과 같은 트리거 지점(체인의 마지막 구간이 완료되는 순간)에서, 결제 성공 여부와 무관하게 `product.status`를 `sold`로 갱신하도록 추가했다(이미 판매완료면 건드리지 않아 멱등). 시스템 메시지로 "OO 상품이 판매완료로 처리됐어요"도 함께 남긴다. curl로 거래 완료 전 `on_sale` → 완료 후 `sold`로 바뀌는 것과 시스템 메시지 두 건(판매완료 처리 + 자동 송금)이 순서대로 남는 것을 확인했다.

### 2.13 안 읽은 메시지 배지 (2026-07-23)
사용자가 "알림 설정은 있는데 알림 받는 칸이 없는데? 채팅 안에 알림 전용 하나 만들까?"라고 지적했다. 확인해보니 `NotificationSetting`(채팅/거래 알림 on-off)은 저장만 될 뿐 그 값을 실제로 참조해서 뭔가를 보여주는 코드가 어디에도 없었고, 심지어 `ChatMessage.read_by`(메시지를 누가 읽었는지 기록하는 M2M 필드)도 모델·시리얼라이저에는 있었지만 아무도 쓰지 않는 죽은 필드였다.

실제 푸시 알림(FCM/APNs) 대신, 이미 있던 `read_by`를 실제로 활용해 **안 읽은 메시지 배지**를 구현했다.

- **백엔드**: `ChatRoomSerializer`에 `unread_count`(SerializerMethodField, `messages.exclude(sender=본인).exclude(read_by=본인).count()`) 추가. `POST /chatrooms/{id}/read/`로 그 방의 안 읽은 메시지를 전부 `read_by`에 추가하는 액션 신설.
- **프론트**: `Chat.jsx`가 방을 열 때(과거분)와 WebSocket으로 새 메시지가 올 때(실시간분) 모두 `markChatRoomRead`를 호출해 읽음 처리한다. `ChatList.jsx`(채팅방 목록)에 방별 안 읽은 개수를 빨간 뱃지로 표시. `BottomNav.jsx`의 "채팅" 탭에는 안 읽은 방이 하나라도 있으면 작은 빨간 점을 표시(경로가 바뀔 때마다 다시 확인).
- **검증(curl)**: 구매자가 메시지 3건 전송 → 판매자 쪽 `unread_count` 3 확인, 구매자 자신은 0(본인 메시지는 안 셈) 확인 → 판매자가 `read/` 호출 → `unread_count` 0으로 확인.

### 2.14 "알림 모음" — 실제 알림 피드로 확장 (2026-07-23)
2.13절 직후 사용자가 "option에 on/off가 지금 의미가 없는거 같아서"라고 다시 지적했다 — 맞는 말이었다. 2.13절의 안 읽은 배지는 `NotificationSetting`(chat_enabled/trade_enabled)을 전혀 참조하지 않아서, 설정을 꺼도 아무 효과가 없었다. 사용자가 제안한 대로 "채팅에서 모든 유저에게 '알림 모음'이라는 채팅을 주고, 채팅 알림은 빨간 원, 거래완료 등 거래 알림은 파란 원으로, on/off로 실제로 켜고 끌 수 있게" 다시 만들었다.

- **모델**: `Notification`(recipient, kind: chat/trade, content, chatroom, is_read) 신설.
- **분류 규칙**: 새로 생성된 `ChatMessage`를 보고 `message_type == 'system'`이면 kind=`trade`(참여/퇴장/인증/판매완료/자동송금 등 거래 상태 이벤트가 전부 시스템 메시지로 남는다는 기존 구조를 그대로 활용), 그 외(text/image)는 kind=`chat`으로 분류하는 `create_notifications_for_message(message)` 함수를 `models.py`에 추가.
- **설정이 실제로 의미를 갖도록**: 이 함수가 알림을 만들기 전에 수신자의 `NotificationSetting`에서 해당 kind(`chat_enabled`/`trade_enabled`)가 꺼져 있으면 아예 생성하지 않는다 — 껐을 때 실제로 그 종류의 알림이 하나도 안 쌓이는 것까지 curl로 확인(판매자가 `trade_enabled=false`로 끈 상태에서 거래를 완료시켜도 그 판매자에게는 trade 알림이 0건, 설정을 안 건드린 구매자는 정상적으로 2건 쌓임).
- **훅 지점 3곳**: ①서버가 직접 만드는 시스템 메시지(`broadcast_system_message` — 참여/퇴장/판매완료/자동송금), ②WebSocket으로 오는 메시지(`ChatConsumer.save_message` — 텍스트/사진 및 프론트가 보내는 인증 완료 시스템 메시지), ③REST 폴백(`POST /chatrooms/{id}/messages`). 세 곳 모두 메시지 생성 직후 `create_notifications_for_message`를 호출하도록 통일.
- **API**: `GET /notifications/feed`(최근 50건 + kind별 안 읽은 개수), `POST /notifications/feed`(전부 읽음 처리).
- **프론트**: `ChatList.jsx` 최상단에 고정된 "알림 모음" 행(🔔) — 빨간 원(채팅)/파란 원(거래) 안 읽은 개수를 함께 표시, 누르면 새 화면 `NotificationFeed.jsx`(내용+시간, kind별 색 점)로 이동하며 열자마자 전부 읽음 처리됨. `BottomNav.jsx`의 "채팅" 탭에도 같은 두 색 점을 표시.
- **검증**: curl로 (1) 일반 메시지 전송 → kind=chat 알림 생성 확인, (2) 거래 완료(2.10/2.12절의 자동 송금+판매완료 시스템 메시지) → kind=trade 알림 생성 확인, (3) 수신자가 trade_enabled를 꺼두면 거래 완료시켜도 그 사람에게는 알림이 안 쌓이는지 확인, (4) 전부 읽음 처리 후 카운트가 0으로 떨어지는지 확인. Playwright로 실제 화면 렌더링까지 확인 — "알림 모음" 행의 빨간 배지, 채팅방 목록의 방별 배지, 하단 네비 빨간 점이 전부 정상적으로 뜨는 것을 스크린샷으로 확인.

### 2.15 실배포 전 필수 보안 조치 3가지 처리 (2026-07-24)
`보안점검.md` 3장 OWASP 점검에서 "지금 당장 손볼 만한 것"으로 남겨뒀던 3가지를 마감 직전 세션에서 처리했다.

- **`SECRET_KEY` 기본값 제거**: `config/settings.py`가 `DJANGO_SECRET_KEY` 환경변수 누락 시 `'dev-only-secret-key'`라는 고정 기본값으로 조용히 기동하던 것을, 값이 없으면 `ImproperlyConfigured`를 던져 기동 자체를 막도록 변경. 고치는 과정에서 로컬 `backend/.env`의 키가 실제로는 `.env.example`의 플레이스홀더(`change-this-secret-key`)가 그대로 남아있던 것을 발견해, `get_random_secret_key()`로 새로 발급했다.
- **로그인/회원가입/송금 rate limiting 추가**: `REST_FRAMEWORK`에 `ScopedRateThrottle`을 전역 기본값으로 넣되(스코프 지정 안 된 뷰는 영향 없음), `SignupView`/`LoginView`에 `throttle_scope='auth'`(10/min, IP 기준), `TransferViewSet`에 `throttle_scope='transfer'`(30/min, 유저 기준)를 적용했다. curl로 같은 IP에서 로그인을 연속 시도하면 11번째부터 `429 Too Many Requests`가 오는 것을 확인했다.
- **배포용 LOGGING 설정 추가**: `settings.py`에 `LOGGING` 자체가 없어 인증 실패/권한 거부가 어디에도 안 남던 문제를 발견했다. `django.security`/`django.request` 로거를 콘솔에 연결하는 것만으로는 DRF가 자체 처리하는 401/403/429가 안 잡혀서, `marketplace/exceptions.py`에 커스텀 `EXCEPTION_HANDLER`를 추가해 그 세 상태코드에 한해 상태코드·유저·경로를 `django.security` 로거로 남기도록 했다.
- **검증**: `python manage.py check` 통과, curl로 로그인 무차별 시도 시 서버 콘솔에 `WARNING django.security: 429 Throttled user=anonymous path=/api/auth/login/`가 실제로 찍히는 것까지 확인.

### 2.16 채팅 UX 다듬기 + 판매자 대리인 초대 지원 (2026-07-24)
사용자가 "3~4인 채팅에서 누가 보낸 메시지인지 구별이 안 간다"고 지적한 것과, 429 에러 문구에 영어가 섞여 보인다는 지적을 처리하면서, 자연스럽게 "판매자는 왜 대리인을 못 부르냐"는 질문으로 이어져 그동안 미구현이던 판매자 대리인 초대를 이번에 구현했다.

- **채팅 발신자 구분**: `Chat.jsx`에서 다른 참여자가 2명 이상(대리인 포함 3~4인 채팅)일 때만 상대방 메시지 말풍선 위에 닉네임을 표시하도록 추가(1:1 채팅은 기존처럼 아바타만). `styles.css`에 `.bubble-col`/`.msg-sender` 추가.
- **429 에러 한글화**: DRF 기본 `Throttled` 메시지("Request was throttled. Expected available in N seconds.")가 그대로 노출되던 것을, `marketplace/exceptions.py`의 `EXCEPTION_HANDLER`가 `Throttled` 예외를 가로채 완전한 한글 문구로 바꾸도록 수정.
- **판매자 대리인 초대**: `ChatParticipant.Role`을 `agent` 단일 값에서 `seller_agent`/`buyer_agent`로 분리(`0007_alter_chatparticipant_role` 마이그레이션, 기존 `agent` 데이터는 전부 `buyer_agent`로 이관). 어느 쪽 대리인인지는 클라이언트가 정하지 않고, `POST /chatrooms/{id}/participants`를 호출한 사람이 그 채팅방에서 판매자인지 구매자인지를 서버가 보고 자동으로 정한다(`ChatParticipantSerializer.role`도 `read_only`로 바꿔 클라이언트가 임의로 role을 지정하는 경로 자체를 막음). 같은 쪽 대리인은 1명으로 제한하고, 대리인은 또 다른 대리인을 초대할 수 없다. `VERIFICATION_ROLE_ORDER`/프론트 `ROLE_ORDER`에 `seller_agent`(1)/`buyer_agent`(2) 순서를 추가해, 초대 순서와 무관하게 인증 체인이 항상 판매자→판매자 대리인→구매자 대리인→구매자로 정렬되게 했다.
- **검증**: curl로 판매자 계정과 구매자 계정이 각자 자기 쪽 대리인을 초대해 4인 채팅을 구성 → 4단계 인증(판매인증→인수인증→인계인증→구매인증)을 순서대로 진행해 각 단계 본인 확인(403 검증 포함)과 최종 `거래확정` 전환, 자동 송금·판매완료 처리까지 정상 동작하는 것을 처음부터 끝까지 확인. 대리인 슬롯이 이미 찬 경우("이미 대리인이 있어요")와 대리인이 대리인을 초대하려는 경우("대리인은 대리인을 초대할 수 없어요") 거부도 확인. `python manage.py check`, `npm run build` 통과.

## 3. 구현 중 발견/해결한 주요 이슈

설계 단계에는 없었지만 구현하면서 드러나 수정한 것들 (설계문서 7.2절 기준).

| 이슈 | 해결 |
|---|---|
| 상품 이미지 업로드/상태 변경에 소유권 검증 누락 | 로그인만 하면 남의 상품도 건드릴 수 있던 취약점 발견 → `IsOwnerOrReadOnly` 권한 클래스 추가 |
| 대면 거래 인증 위조 가능 | 클라이언트가 보낸 `from_user`/`to_user`를 그대로 신뢰 → 서버가 참여자 목록에서 체인을 직접 계산하도록 변경 (2.3절 3번) |
| 사진 순서 변경 시 `IntegrityError` | `(product_id, order)` 유니크 제약 때문에 두 사진 순서를 바로 맞바꾸면 충돌 → 임시로 1만 이상 값으로 옮긴 뒤 최종 순서를 저장하는 2단계 갱신으로 처리 |
| WebSocket이 커스텀 인증 헤더를 못 보냄 | 토큰을 쿼리스트링(`?token=`)으로 전달받아 인증하는 미들웨어(`ws_auth.py`)를 별도 구현 |
| 가격 체계 미설계 | 송금/에스크로 요구사항이 있었지만 `products`에 price 컬럼이 없어, 결제 PG 연동 없이는 의미 있는 구현이 어렵다고 판단해 이번 범위에서 보류 |

## 4. 현재 상태 요약
