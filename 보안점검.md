# 보안 점검 기록 (2026-07-22)

`securecondhand`를 GitHub에 올린 뒤 붙인 자동 취약점 스캔 도구와, 그 결과로 나온 의존성 업데이트를 실제로 검토·테스트·반영한 기록이다. 5장은 이후 추가한 송금 기능에 대한 별도 보안 점검이다.

## 1. 붙인 도구

| 도구 | 역할 | 설정 파일 |
|---|---|---|
| **GitHub CodeQL (Code Scanning)** | 코드 자체의 취약점(SAST) — push/PR마다 + 매주 월요일 자동 스캔 | `.github/workflows/codeql.yml` |
| **Dependabot** | `package.json`(npm), `backend/requirements.txt`(pip), GitHub Actions의 알려진 취약 버전을 찾아 업데이트 PR을 자동 생성 | `.github/dependabot.yml` |

두 도구 모두 GitHub 네이티브 기능이라 별도 계정/설치 없이 저장소 Settings만으로 동작한다. 결과는 저장소의 **Security → Code scanning alerts**(CodeQL)와 **Pull requests**(Dependabot) 탭에서 확인한다.

## 3. OWASP Top 10 (2021) 체크리스트

의존성 취약점과 별개로, 코드/설정을 직접 훑어서 OWASP Top 10 각 항목에 이 프로젝트가 어떻게 대응하고 있는지 점검했다. ✅ 대응됨 / ⚠️ 부분적·확인 필요 / ❌ 미흡.

| 항목 | 상태 | 근거 / 남은 조치 |
|---|---|---|
| **A01 Broken Access Control** | ✅ | 상품 수정/삭제/이미지/상태변경에 `IsOwnerOrReadOnly`, 관리자 API에 `IsStaffUser` 적용. 친구 요청 수락은 서버가 `receiver == request.user`인지 검증. 대면 거래 인증도 클라이언트가 보낸 `from_user`/`to_user`를 안 믿고 서버가 참여자 목록에서 직접 체인을 계산해 검증(`views.py`, 개발과정.md 2.3절). 송금도 `sender`를 클라이언트가 지정 못 하고 항상 `request.user`로 서버가 강제(5장 참고). |
| **A02 Cryptographic Failures** | ⚠️ | 비밀번호는 Django 기본 해셔(PBKDF2)로 저장돼 평문 노출 없음. **다만 `config/settings.py`의 `SECRET_KEY = os.getenv('DJANGO_SECRET_KEY', 'dev-only-secret-key')`가 환경변수 누락 시 고정된 약한 기본값으로 조용히 fallback한다 — 배포 시 `.env` 설정을 깜빡하면 예측 가능한 키로 뜬다.** 실서비스 전환 전 HTTPS(WSS) 적용 필요(중고거래플랫폼_설계문서.md 7.4절에 별도 기재). |
| **A03 Injection** | ✅ | 전 구간 Django ORM만 사용 (`.raw()`/`cursor.execute()`/`.extra()` 사용 없음 — 코드베이스 전체 grep으로 확인). 입력값은 DRF serializer가 검증. |
| **A04 Insecure Design** | ✅ | 대면 거래 인증 위조 방지(A01 참고), 상품 사진 순서 변경 시 유니크 제약 충돌을 막기 위해 임시값 경유 2단계 갱신으로 설계(설계문서 7.2절) — 둘 다 "구현하다 발견해서 설계를 바꾼" 사례. 송금의 동시성/이중지불 방지 설계도 동일한 맥락(5장 참고). |
| **A05 Security Misconfiguration** | ⚠️ | `DEBUG`/`ALLOWED_HOSTS`/`CORS_ALLOWED_ORIGINS`는 환경변수로 분리돼 있어 양호. 다만 `LOGGING` 설정이 아예 없고, `SECURE_SSL_REDIRECT`/`SESSION_COOKIE_SECURE`/`CSRF_COOKIE_SECURE`/`CSRF_TRUSTED_ORIGINS` 등 배포용 보안 헤더 설정도 없음 — 로컬 개발 단계라 당장은 아니지만 실배포 직전 반드시 채워야 함. |
| **A06 Vulnerable and Outdated Components** | ✅ | CodeQL + Dependabot 상시 가동 (본 문서 1~2장). 최초 스캔에서 나온 12건 전부 검증 후 반영. |
| **A07 Identification and Authentication Failures** | ⚠️ | DRF `TokenAuthentication` + WebSocket 쿼리스트링 토큰 인증(`ws_auth.py`) 사용. 비밀번호 변경 시 기존 토큰을 폐기하고 새 토큰을 발급해 세션 고정(session fixation) 문제를 막음. **다만 로그인/회원가입 엔드포인트에 rate limiting이 없다** (`REST_FRAMEWORK`에 `DEFAULT_THROTTLE_CLASSES` 미설정) — 무차별 대입(brute force) 공격에 노출될 수 있음. `django-ratelimit`이나 DRF `Throttle` 클래스 적용 권장. |
| **A08 Software and Data Integrity Failures** | ✅ | `package-lock.json`/`requirements.txt`로 의존성 버전이 고정돼 빌드마다 다른 코드가 딸려오지 않음. 배포 파이프라인이 없어(수동 배포) CI 단계의 코드 변조 위험도 해당 없음. |
| **A09 Security Logging and Monitoring Failures** | ❌ | `settings.py`에 `LOGGING` 설정이 없어, 인증 실패나 403(권한 거부) 같은 보안 이벤트가 별도로 기록되지 않고 `runserver` 콘솔 출력에만 의존한다. 실배포 시 최소한 인증 실패/관리자 액션 로깅 추가 필요. |
| **A10 Server-Side Request Forgery (SSRF)** | ✅ 해당 없음 | 사용자가 준 URL을 서버가 대신 요청(fetch)하는 기능 자체가 없음 (이미지는 업로드 방식이지 URL 입력 방식이 아님). |

**정리하면 지금 당장 손볼 만한 건 3가지**: ① `SECRET_KEY` 기본값 제거(환경변수 없으면 명시적으로 죽게), ② 로그인/회원가입에 요청 제한(rate limit) 추가, ③ 배포용 `LOGGING` 설정 추가. 셋 다 로컬 개발 단계에서는 치명적이지 않지만 실서비스 전환 전에는 반드시 처리해야 한다.

## 4. 송금 기능 보안 점검 (2026-07-22 추가)

과제 최소 요구사항인 "유저 간 송금"을 내부 잔액 시스템으로 구현하면서(`개발과정.md` 2.5절), 돈이 오가는 기능 특유의 위험 요소를 별도로 점검했다.

| 위험 | 점검 내용 | 결과 |
|---|---|---|
| 잔액 이중지불/음수화 (동시성) | `POST /transfers`가 sender/receiver의 `Profile` 행을 `select_for_update()`로 잠그고 같은 트랜잭션 안에서 검증→차감→입금을 수행하는지 | ✅ curl로 잔액 990,000원에서 200,000원씩 10개 동시 요청 → 정확히 4건만 성공(800,000원 차감), 5번째부터는 잔액부족으로 거부, 잔액은 절대 음수로 내려가지 않음을 확인 |
| 교착상태(deadlock) | 두 사람이 동시에 서로에게 송금할 때 잠그는 순서가 어긋나지 않는지 | ✅ 잠금 순서를 항상 `user_id` 오름차순으로 고정 |
| sender 위조 (IDOR) | 클라이언트가 `sender`를 임의로 지정해 남의 잔액에서 빼낼 수 있는지 | ✅ `sender`는 serializer에 없고 서버가 `request.user`로만 생성 |
| 잔액 직접 조작 | `PATCH /profiles/me`로 `balance`를 직접 바꿀 수 있는지 | ✅ `ProfileSerializer`에서 `balance`를 `read_only_fields`로 지정, PATCH 요청으로 실제 반영 안 됨을 curl로 확인 |
| 자기 자신에게 송금 | 셀프 송금으로 잔액을 부풀리거나 로직을 악용할 수 있는지 | ✅ 애플리케이션 레벨(수신자 조회 시 본인 제외)과 DB 체크 제약(`prevent_self_transfer`) 이중으로 차단 |
| 음수/0원 송금 | 마이너스 금액으로 상대 잔액을 빼가거나 무의미한 0원 요청을 보낼 수 있는지 | ✅ `PositiveIntegerField`(모델) + `validate_amount`(serializer)로 이중 차단, curl로 확인 |
| 사용자 열거(enumeration) | 존재하지 않는 아이디와 "본인 아이디"를 구분해 응답하면 아이디 존재 여부를 유추할 수 있는지 | ✅ 두 경우 모두 동일한 오류 메시지("본인이 아닌 대상 사용자를 찾을 수 없어요") 반환 |
| 원장 위변조 | 생성된 송금 기록을 나중에 수정/삭제해 거래 내역을 조작할 수 있는지 | ✅ `TransferViewSet.http_method_names`에서 PATCH/DELETE 자체를 제외(405 확인) — 원장 불변 |

남은 조치(기존 A07 항목과 동일선상): 송금 API에도 별도 rate limiting은 아직 없다. 소액을 짧은 시간에 반복 요청하는 어뷰징까지 막으려면 DRF 스로틀 클래스 적용이 필요하나, 실배포 전 처리 항목으로 남겨둔다.

## 5. 악성 유저·상품 자동 차단 보안 점검 (2026-07-22 추가)

과제 Group A 필수 항목인 "일정 횟수 이상 신고 시 자동 차단/휴면 전환"이 코드에 빠져 있던 것을 발견해 구현하면서(`개발과정.md` 2.6절), 신고 기능 자체가 공격 도구로 악용될 수 있는 경로를 함께 점검했다.

| 위험 | 점검 내용 | 결과 |
|---|---|---|
| 신고 기능의 무기화 (허위 신고로 경쟁자 차단) | 신고 1건이 아니라 "서로 다른 신고자 수"로 임계값을 세는지, 즉 신고자 1명이 여러 번 신고를 반복해서 혼자 임계값을 채울 수 있는지 | ✅ 같은 신고자가 동일 대상에 5번 반복 신고해도 차단되지 않음을 curl로 확인. `values('reporter').distinct().count()`로 서로 다른 신고자만 집계 |
| 정상 임계값 동작 | 서로 다른 신고자 3명이 모이면 실제로 제재되는지 | ✅ 상품: 3명이 신고 후 `GET /products/{id}` 200 → 404로 전환 확인. 유저: 3명이 신고 후 로그인 200 → 400(계정 비활성)으로 전환 확인 |
| 관리자 조회 가능성 유지 | 차단된 상품/휴면 유저를 관리자가 여전히 조회·해제할 수 있는지 (자동 차단이 곧 영구 삭제가 되지 않는지) | ✅ `AdminProductViewSet`/`AdminUserViewSet`은 `is_blocked`/`is_active` 필터 없이 전체 조회 — 관리자 계정으로 차단 상품도 그대로 응답됨을 확인 |
| 소유자 우회 | 상품 소유자가 자기 차단된 상품을 여전히 노출/수정할 수 있는지 | ✅ `ProductViewSet.get_queryset`이 스태프가 아니면 소유자 포함 전원에게서 `is_blocked=True` 상품을 제외 |

남은 조치: 지금은 "신고 3건(서로 다른 신고자)"이면 즉시 자동 제재되고 별도의 이의제기(해제 요청) 절차는 없다. 실서비스 전환 시에는 오탐(정당한 상품이 몰이 신고로 차단되는 경우)에 대비한 이의제기 플로우를 추가하는 것이 바람직하다.

## 6. 토스페이먼츠 지갑 충전 보안 점검 (2026-07-22 추가)

사용자 요청으로 지갑 충전에 실제 PG 테스트 연동(`개발과정.md` 2.7절)을 붙이면서, 돈이 실제로(비록 테스트지만) PG 서버를 거쳐 들어오는 구조 특유의 위험을 점검했다.

| 위험 | 점검 내용 | 결과 |
|---|---|---|
| 결제 금액 위조 | 클라이언트가 결제창 호출 시 사용한 금액과 다른(더 큰) 금액을 승인 요청에 실어 보내 잔액을 부풀릴 수 있는지 | ✅ 서버가 주문 생성 시 기록해둔 금액과 승인 요청의 금액을 대조해, 다르면 토스 서버에 확인하기조차 하지 않고 400으로 차단함을 curl로 확인 |
| 승인 없이 잔액만 조작 (클라이언트 신뢰) | 프론트가 리다이렉트됐다는 사실만으로 잔액을 올려주는지, 아니면 서버가 토스와 직접 통신해 진짜 승인 여부를 확인하는지 | ✅ 가짜 `payment_key`로 승인 요청 → 실제 토스 서버가 "결제 진행 데이터 없음" 오류를 반환하고 주문이 `failed`로 전환되며 잔액은 그대로임을 확인. 클라이언트 리다이렉트만으로는 절대 잔액이 오르지 않음 |
| 중복 승인/중복 적립 | 성공 페이지를 새로고침하거나 승인 요청을 두 번 보내면 잔액이 두 번 적립되는지 | ✅ 주문이 이미 `confirmed`면 재요청해도 다시 적립하지 않고 기존 값만 반환함을 확인(멱등성) |
| 시크릿 키 노출 | `TOSS_SECRET_KEY`가 프론트 번들이나 API 응답에 노출되는지 | ✅ `GET /payments/config`는 `client_key`만 내려주고, 시크릿 키는 백엔드 `.env`에서만 읽어 서버 간 호출에만 사용 (grep으로 프론트 코드에 시크릿 키 참조 없음을 확인) |
| 남의 주문으로 승인 시도 (IDOR) | 다른 사람이 만든 주문의 `order_id`로 내가 승인 요청을 보낼 수 있는지 | ✅ `PaymentOrder.objects.filter(order_id=, user=request.user)`로 조회하므로 본인 주문이 아니면 404 |

**미검증 항목**: 실제 브라우저에서 결제창을 열고 토스 테스트 카드번호로 결제를 완료하는 성공 경로 전체는 브라우저 환경이 없어 이 세션에서 직접 확인하지 못했다(코드 검증은 시뮬레이션된 승인 데이터로 진행). 사용자가 `/wallet/topup`에서 직접 한 번 시도해봐야 한다.

## 7. 결론

- 저장소에 **CodeQL + Dependabot**을 상시 가동 상태로 붙였다. 앞으로 push/PR마다 코드 취약점 스캔이, 매주 의존성 취약점 점검이 자동으로 돈다.
- Dependabot이 처음 연 12건 전부 실제로 설치·빌드·기동까지 검증한 뒤 **전부 반영 완료**했다.
- 그 과정에서 Django 6 비호환 코드(`CheckConstraint`)를 실제로 발견해 고쳤고, vite 8이 요구하는 Node.js 20+ 미충족 문제는 **nvm으로 Node 20을 설치해 인프라 쪽을 해결**했다 (`package.json`에 `engines` 필드로 명시).
