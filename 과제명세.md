## Tiny Second-hand Shopping Platform

- 플랫폼의 요구사항을 만족하도록 개발
- 플랫폼 개발 전 과정(요구사항 분석, 시스템 설계, 구현, 체크리스트 작성 및 테스팅, 유지보수)에 대한 보고서 작성(양식은 자유)
- 개발 과정에서 확인한 보안 약점들이 무엇이고, 어떻게 변경하였는지 작성
- 완성한 프로그램은 github에 public으로 올리고, README.md에 환경 설정 및 실행방법 명시
- 다음 페이지에서 언급한 최소 요구사항을 모두 만족한다면, 이외 어떤 기능이든 추가 가능하며, 다른 모든 것이 자유.
- 단, 최대한 보안 약점이 없도록 할 것.
- ChatGPT, Copilot, Cursor, Claude 등 AI 도구를 최대한 적극적으로 활용할 것!

## 최소 요구사항 (슬라이드 35페이지)

중고거래 플랫폼에 있어야 하는 기능은 두 그룹으로 나뉜다. 요구사항 추가/논리적으로 타당한 변경은 자유.

| 그룹 | 항목 | 지시사항 | 현재 상태 |
|---|---|---|---|
| A | 회원가입, 상품 등록/조회, 사용자 간 소통(채팅), 악성 유저·상품 차단(신고) | 시스템 설계는 **설계문서 24페이지 상당 명세**(이 저장소의 `중고거래플랫폼_설계문서.md` 1~2장)를 모두 포함할 것 (필요 시 기능 추가) | ✅ 구현 완료 (`개발과정.md`, 설계문서 7.1절) |
| B | 유저 간 송금, 상품 검색, 관리자 전체 관리 | **시스템 분석부터 새로 진행할 것** | 검색 ✅ / 관리자 ✅ / **송금 ✅ 구현 완료** |

- 상품 검색: `GET /products?q=&category=&trade_type=&sort=` 로 구현됨 (설계문서 5장).
- 관리자: `/admin` 대시보드 + `IsStaffUser` 전용 API로 구현됨 (설계문서 5장, 7.1절).
- **유저 간 송금**: `products`에 가격(price) 필드가 없어 미구현이던 상태였으나(설계문서 7.3절), 실제 결제 PG 연동 없이 **내부 포인트/잔액 기반 송금**(`POST /api/transfers/`)으로 범위를 좁혀 구현 완료. 상품에 `price`, 프로필에 `balance`를 추가하고 `Transfer` 원장 모델을 신설했다 — 자세한 내용은 설계문서 7장, `개발과정.md`, `보안점검.md` 참고. (에스크로/보류결제까지는 범위 밖으로 유지)
  - 추가로 **토스페이먼츠 테스트 연동**(사업자등록 불필요, 개발자 가입만으로 발급되는 테스트 키)으로 "지갑 충전" 화면을 붙였다. PG 결제는 가맹점 계좌로 들어오지 특정 개인에게 바로 가지 않으므로, 유저 간 송금 자체를 대체하진 못하고 "충전 → 이후 내부 잔액으로 송금"하는 구조다 (`개발과정.md` 2.7절, `보안점검.md` 6장).
- **악성 유저·상품 차단**: 설계문서 25p(2.5절 상당, 시스템 설계 상세)에 "일정 횟수 이상 신고된 상품은 차단, 유저는 휴면계정 전환"이 명시돼 있었지만, 코드 확인 결과 신고 접수/관리자 수동 처리만 있고 **자동 차단 로직 자체가 빠져 있었음** (Group A 필수 항목인데도 누락). 서로 다른 신고자 3명 이상이 같은 대상을 신고하면 상품은 자동 차단(목록/상세에서 숨김), 유저는 자동 휴면(로그인 차단)되도록 이번 세션에서 구현·검증 완료 (`개발과정.md` 2.6절, `보안점검.md` 4장).
