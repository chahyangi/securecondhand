# 중고거래 플랫폼 설계 문서

소프트웨어 개발 주기: 요구사항 도출 → 시스템 설계 → 시스템 구현 → 테스팅 → 유지보수
본 문서는 요구사항 도출 ~ 시스템 설계(화면 설계 / DB 설계 / 아키텍처 / API 설계)까지의 산출물을 정리한 것입니다.

**플랫폼**: 모바일 웹 전용. 화면은 최대 폭 480px의 고정 모바일 프레임(`app-shell`)으로 렌더링되며, 데스크톱 브라우저에서도 화면 중앙에 동일한 모바일 폭으로 표시된다. 데스크톱 전용 레이아웃은 제공하지 않는다.

---

## 1. 요구사항 도출

### 1.1 기능적 요구사항

#### 회원/계정
- 회원가입, 로그인/로그아웃 (이메일, 소셜 로그인)
- 프로필 관리 (닉네임, 프로필 아이콘)
- 비밀번호 찾기, 회원 탈퇴

#### 친구 기능
- 친구 추가/삭제, 친구 목록 조회
- 친구 요청 발신/수신 관리

#### 대리인 기능
- 채팅창 내에서 친구 중 1명을 **일회성으로 초대**하여 거래 대행
- 대리인 포함 시 채팅방이 3인 또는 4인 채팅으로 확장
- **거래 완료 시 대리인 권한 자동 해제 및 채팅방 자동 퇴장**

#### 상품 등록 및 조회
- 상품 등록 시 사진 필수 업로드 (다중 이미지)
- 사진 순서 변경, 대표 사진 지정
- 제목, 카테고리, 상태(미개봉 / 개봉후 미사용 / 사용감 있음), 거래방식(직거래/택배) 입력
- 등록일 + 최종 수정일 기록
- 상품 수정/삭제, 판매완료/예약중 상태 변경

#### 메인 화면 (핵심 차별점)
- 텍스트 리스트가 아닌 **사진 그리드(바둑판식)** 로 상품 나열
- 통일된 비율(정사각형)로 자동 크롭/리사이징
- 그리드 셀은 **사진만** 표시 (텍스트/가격/아이콘 없음)
- 무한 스크롤
- 모바일 고정 폭(480px) 기준 3열 그리드
- 필터/정렬은 검색창 내에서 적용 (메인 화면에는 별도 필터바 없음)

#### 검색
- 키워드 검색 (제목, 설명 기반)
- 검색창 내 카테고리/지역/거래방식 필터, 정렬(최신순/인기순)
- 최근 검색어

#### 소통 (채팅)
- 사용자 간 채팅 (대리인 포함 시 최대 4인)
- 채팅 목록, 읽음 표시, 알림
- 채팅 내 사진 전송

#### 거래/송금
- 사용자 간 송금
- 거래 내역 조회
- 안전결제(에스크로) 옵션

#### 대면 거래 인증
- 대면 거래 시 양측 사용자가 각자 확인 버튼을 눌러 실거래 여부를 수동으로 확인
- 확인 완료 시 거래 상태 자동 전환("거래확정")

#### 신뢰/안전
- 신고 기능 (사용자/상품)
- 관리자 검토 후 제재
- 금지어/이미지 필터링
- 특정 유저 차단

#### 관심/알림
- 찜하기 (상품상세 내 위치)
- 채팅, 거래 상태 알림

#### 관리자 기능
- 전체 사용자/상품 관리
- 신고 처리 대시보드
- 통계 대시보드
- 공지사항/이벤트 관리

### 1.2 비기능적 요구사항
- **보안**: 로그인 인증, 개인정보 암호화, 결제 정보 보호, 파일 업로드 검증, 대리인 권한 위·변조 방지
- **디자인/UX**: 사진 위주 그리드형 UI 일관성, 모바일 웹 고정 폭(480px) 레이아웃
- **성능**: 이미지 썸네일 최적화, lazy loading, 무한 스크롤 시 끊김 없는 로딩
- **확장성/가용성**: 서버·스토리지 확장성, 다운타임 최소화
- **스토리지 비용**: 이미지 압축, CDN 활용
- **법적 요소**: 개인정보처리방침, 전자상거래 법규 준수

## 2. 화면 설계

화면은 총 4개로 구성: **메인 / 상품상세 / 채팅 / 설정**

### 2.1 메인화면

| 영역 | 내용 |
|---|---|
| 상단바 | 로고, 검색창(필터 포함), 프로필 아이콘 |
| 사진 그리드 | 바둑판식 정사각형 썸네일, 무한 스크롤 |
| 그리드 셀 | 사진만 |
| 하단 네비게이션 | 홈 / 검색 / 채팅 / 설정 |

**인터랙션**
- 그리드 셀 클릭 → 상품상세 이동
- 검색창 클릭/입력 → 카테고리/정렬/지역 등 필터 적용
- 스크롤 하단 도달 → 다음 페이지 자동 로드

### 2.2 상품상세

| 영역 | 내용 |
|---|---|
| 상단바 | 뒤로가기, 신고 아이콘 |
| 사진 영역 | 다중 사진 슬라이드, 대표 사진부터 |
| 상품 정보 | 제목, 카테고리, 상태(미개봉/개봉후미사용/사용감있음), 거래방식, 등록일, 최종 수정일 |
| 설명 | 상품 설명 텍스트 |
| 판매자 정보 | 프로필 아이콘 + 닉네임 (지역 없음) |
| 액션 바 | 찜 아이콘, 채팅하기 버튼 |

**인터랙션**
- 사진 스와이프, 찜 토글, 채팅방 이동, 판매자 프로필 이동, 신고

**상태 표시**
- 판매완료/예약중 시 배지 표시, 채팅하기 버튼 비활성화/문구 변경

### 2.3 채팅

| 영역 | 내용 |
|---|---|
| 상단바 | 뒤로가기, 참여자 아이콘(2~4인), 상품 미니 배너 |
| 메시지 영역 | 텍스트/사진 말풍선, 시간, 읽음 표시 |
| 대리인 초대 버튼 | 친구 목록에서 1명 선택, 일회성 초대 |
| 대면 거래 확인 영역 | 거래 확인 버튼, 상태 표시(대기중/확인됨) |
| 입력창 | 텍스트/사진 첨부, 전송 |

**인터랙션**
- 대리인 초대 → 채팅방 3인/4인으로 확장
- 거래완료 전환 시 대리인 자동 퇴장 + 시스템 메시지
- 양측 확인 완료 시 거래 상태 자동 "거래확정" 전환

### 2.4 설정

| 하위 화면 | 내용 |
|---|---|
| 프로필 관리 | 아이콘 변경, 닉네임 수정, 계정 정보 확인 |
| 친구 관리 | 친구 목록, 추가/삭제, 요청함 |
| 알림 설정 | 채팅/거래 상태 알림 on/off |
| 계정 관리 | 비밀번호 변경, 회원 탈퇴 |
| 찜 목록 | 찜한 상품 목록 조회, 개별 찜 해제 |
| 신고 내역 | 내가 신고한 내역과 처리 상태 조회 |

> 대리인은 채팅창 내 일회성 초대 방식이므로 설정 화면에는 별도 "대리인 설정" 메뉴 없음.

## 3. DB 설계 (PostgreSQL)

보안과 데이터 정합성을 위해 DB는 PostgreSQL을 사용한다. 인증 기본 정보는 Django의 `auth_user`를 사용하고, 서비스별 데이터는 별도 테이블로 분리한다.

### profiles
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 프로필 ID |
| user_id | FK(auth_user) unique | Django 계정 |
| nickname | varchar(40) | 닉네임 |
| icon | varchar(255) | 프로필 아이콘/이미지 경로 |
| balance | int (unsigned) | 내부 포인트성 잔액. 가입 시 기본 1,000,000 지급, `Transfer` 처리 시에만 증감 (API로 직접 수정 불가) |
| created_at / updated_at | timestamptz | 생성/수정 시각 |

### friend_requests
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 친구 요청 ID |
| requester_id | FK(auth_user) | 요청자 |
| receiver_id | FK(auth_user) | 수신자 |
| status | varchar(20) | pending / accepted / rejected |
| created_at / updated_at | timestamptz | 생성/수정 시각 |

### categories
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 카테고리 ID |
| name | varchar(40) unique | 카테고리명 |

### products
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 상품 ID |
| seller_id | FK(auth_user) | 판매자 |
| category_id | FK(categories) | 카테고리 |
| title | varchar(120) | 제목 |
| condition | varchar(20) | new / opened / used |
| trade_type | varchar(20) | direct / delivery |
| description | text | 설명 |
| price | int (unsigned) | 가격(원). 송금 기능 도입과 함께 추가 |
| status | varchar(20) | on_sale / reserved / sold |
| is_blocked | boolean | 신고 누적(서로 다른 신고자 3명 이상)으로 자동 차단됨. 기본 false |
| created_at / updated_at | timestamptz | 생성/수정 시각 |

### product_images
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 이미지 ID |
| product_id | FK(products) | 상품 |
| image | varchar/file path | 업로드 이미지 |

| is_representative | boolean | 대표 사진 여부 |
| created_at / updated_at | timestamptz | 생성/수정 시각 |

### wishlists
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 찜 ID |
| user_id | FK(auth_user) | 사용자 |
| product_id | FK(products) | 상품 |
| created_at | timestamptz | 생성 시각 |

### chatrooms / chat_participants / chat_messages
| 테이블 | 핵심 컬럼 | 설명 |
|---|---|---|
| chatrooms | product_id, status | 상품별 채팅방, 거래 상태 |

### trade_verifications
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 인증 ID |
| chatroom_id | FK(chatrooms) | 채팅방 |
| verifier_a_id / verifier_b_id | FK(auth_user) | 확인 참여자 |
| verification_type | varchar(20) | handover / trade |
| status | varchar(20) | pending / manual |
| verified_at | timestamptz nullable | 인증 완료 시각 |

### verification_steps
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 단계 ID |
| chatroom_id | FK(chatrooms) | 채팅방 |
| user_id | FK(auth_user) | 버튼을 누른 사용자 |
| hop_index | smallint | 인증 체인 내 구간 순번 |
| side | varchar(10) | give(인계) / receive(인수) |
| 유니크 제약 | (chatroom_id, hop_index, side) | 구간·방향당 1건만 존재 |

`trade_verifications`가 "구간이 완료됐다"는 최종 기록이라면, `verification_steps`는 그 구간을 이루는 give/receive 버튼 각각을 눌렀는지 기록해 새로고침해도 진행 상태가 유지되게 한다. 양쪽이 다 채워지면 `trade_verifications` 레코드가 생성/갱신된다.

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 설정 ID |
| user_id | FK(auth_user) unique | 사용자 |
| chat_enabled | boolean | 채팅 알림 on/off |
| trade_enabled | boolean | 거래 상태 알림 on/off |

### reports
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 신고 ID |
| reporter_id | FK(auth_user) | 신고자 |
| target_type | varchar(20) | user / product |
| target_id | bigint | 신고 대상 ID |
| reason | text | 신고 사유 |
| status | varchar(20) | pending / resolved |
| created_at / updated_at | timestamptz | 생성/수정 시각 |

### transfers

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 송금 ID |
| sender_id | FK(auth_user) | 보낸 사람 |
| receiver_id | FK(auth_user) | 받은 사람 |
| amount | int (unsigned, >0) | 송금액. DB 제약(`amount__gt=0`)으로도 0/음수 차단 |
| memo | varchar(200) | 메모 (선택) |
| chatroom_id | FK(chatrooms) nullable | 채팅방에서 보낸 경우 해당 거래 맥락 연결 |
| created_at | timestamptz | 생성 시각 |
| 제약 | `sender != receiver` (DB 체크 제약) | 본인에게 송금 금지 |

`transfers`는 생성 후 수정·삭제하지 않는 불변 원장이다(API도 update/delete를 노출하지 않음). 실제 금액 이동은 `sender.profile.balance`/`receiver.profile.balance` 증감으로 이뤄지고, 이 테이블은 그 이동 기록만 남긴다.

### payment_orders

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | bigserial PK | 주문 ID |
| order_id | varchar(64) unique | 토스페이먼츠에 넘기는 주문번호 (UUID) |
| user_id | FK(auth_user) | 충전한 사람 |
| amount | int (unsigned, >0) | 충전 금액. 결제창을 열기 전에 서버가 먼저 기록해두는 값 |
| status | varchar(20) | pending / confirmed / failed |
| payment_key | varchar(200) | 토스가 발급한 결제 키 (승인 완료 후 저장) |
| created_at / updated_at | timestamptz | 생성/수정 시각 |

토스페이먼츠 결제위젯(테스트 연동)으로 **지갑을 충전**하는 주문 기록. 결제 승인(`POST /payments/orders/confirm`) 시 토스가 돌려준 금액을 이 레코드의 `amount`와 대조해 위조를 막고, `status`가 이미 `confirmed`면 다시 잔액을 적립하지 않는다(멱등성). 유저 간 송금(`transfers`)과는 별개 테이블이다 — PG 결제는 가맹점 계좌로 들어오므로 "충전"까지만 대체하고, 충전된 잔액을 유저끼리 주고받는 건 여전히 `transfers`가 담당한다.

### 설계 포인트
| 결정 | 이유 |
|---|---|
| ProductImage → 별도 테이블 | 이미지 순서/대표 사진 변경, 파일 업로드 관리에 유리 |
| ChatParticipant → 별도 테이블 | 대리인 자동 퇴장, 역할/퇴장 시각 관리에 유리 |
| ChatMessage → 별도 테이블 | 메시지가 계속 증가하므로 독립 인덱싱/페이지네이션 필요 |
| FriendRequest, Wishlist, Report → 별도 테이블 | N:M 관계와 로그성 데이터의 정합성 보장 |

**권장 인덱스**: `products.seller_id`, `products.category_id`, `products.status`, `products.created_at`, `chat_messages.{chatroom_id,created_at}`, `wishlists.{user_id,product_id}`(복합 유니크), `transfers.{sender_id,-created_at}`, `transfers.{receiver_id,-created_at}`

## 4. 시스템 아키텍처

### 구성 요소
1. **클라이언트(React/Vite)**: 사진 그리드/채팅 UI 렌더링
2. **백엔드 API 서버(Django + Django REST Framework)**: REST API(회원/상품/검색/찜/친구/신고/관리자) + WebSocket 서버(실시간 채팅, 알림)
3. **데이터베이스**: PostgreSQL
4. **이미지 스토리지 + CDN**: 업로드 → 리사이징/압축 → 저장 → CDN 캐싱
5. **외부 연동**: 푸시 알림(FCM/APNs), 송금/결제 PG, 소셜 로그인(OAuth)

### 데이터 흐름
```
클라이언트 A                       클라이언트 B
      ↓ REST / WebSocket                ↓ REST / WebSocket
              백엔드 API 서버
     ↓              ↓                ↓
 데이터베이스   이미지 스토리지/CDN   외부 서비스(푸시/결제/OAuth)
```

| Method | Endpoint | 설명 |
|---|---|---|

### 지갑 충전 (토스페이먼츠 테스트 연동)
| Method | Endpoint | 설명 |
|---|---|---|
| GET | /payments/config | 공개 client key 제공 (프론트가 SDK 초기화에 사용, secret key는 절대 포함 안 함) |
| GET | /payments/orders | 내 충전 주문 내역 |
| POST | /payments/orders | 충전 주문 생성 (`amount`) — 결제창을 열기 전에 서버가 먼저 금액을 기록 |
| POST | /payments/orders/confirm | 결제 승인 처리 (`order_id`, `payment_key`, `amount`) |

`POST /payments/orders/confirm`은 (1) 요청의 `amount`가 주문 생성 시 기록해둔 금액과 다르면 토스 서버에 확인하기 전에 즉시 거부하고, (2) 일치하면 토스의 결제 승인 API(`POST https://api.tosspayments.com/v1/payments/confirm`, secret key로 Basic 인증)를 서버 간 호출해 실제로 결제가 이뤄졌는지 검증한 뒤에만 `Profile.balance`를 충전한다. 이미 `confirmed`인 주문을 다시 요청해도 잔액을 중복 적립하지 않는다(멱등성). 이 흐름으로 실제 대체되는 건 "지갑 충전"뿐이며, 유저 간 송금은 여전히 위 `/transfers`(내부 잔액 이동)로 처리된다 — PG 결제는 가맹점 계좌로 들어오지 특정 개인에게 바로 가지 않기 때문이다.

## 6. 실행 방법 (개발 환경)

프로젝트는 세 가지 요소로 구성된다: **① PostgreSQL DB → ② Django 백엔드 API → ③ React(Vite) 프론트엔드**.
세 요소는 이 순서로 켜야 하며(DB가 떠 있어야 백엔드가 붙고, 백엔드가 떠 있어야 프론트엔드가 실제 데이터를 받아온다), 백엔드가 꺼져 있으면 프론트엔드는 `src/mock/data.js`의 임시 데이터로 자동 대체된다.

### 한 번에 켜기

세 요소를 순서대로 띄워주는 스크립트가 준비되어 있다 (최초 1회 `migrate`는 별도로 해야 함).

```bash
npm start
# 또는
./start.sh
```

DB(Docker) → 백엔드 → 프론트엔드 순으로 기동하고, 터미널에서 `Ctrl + C` 한 번이면 백엔드·프론트엔드가 함께 종료된다 (Postgres 컨테이너는 계속 떠 있으며, 내리려면 아래 ①의 `docker compose stop postgres`를 따로 실행).

### 한 번에 끄기

터미널을 그냥 닫아버려서 `Ctrl + C`가 안 먹었거나, Postgres 컨테이너까지 한 번에 내리고 싶을 때 쓰는 스크립트.

```bash
npm stop
# 또는
./off.sh
```

Django 백엔드·React 프론트엔드 프로세스를 찾아서 종료하고, Postgres 컨테이너도 `docker compose stop`으로 정지한다(데이터는 보존, 완전히 지우려면 `docker compose down`). 이미 다 꺼져 있는 상태에서 실행해도 안전하다.

### ① PostgreSQL (Docker)

```bash
# 켜기 (백그라운드 실행)
docker compose up -d postgres

# 상태 확인
docker ps

# 끄기 (컨테이너만 정지, 데이터 보존)
docker compose stop postgres

# 완전히 내리기 (컨테이너 삭제, 볼륨의 데이터는 유지됨)
docker compose down
```

### ② Django 백엔드

```bash
cd backend
source .venv/bin/activate

# 최초 1회 또는 모델 변경 후
python manage.py migrate

# 켜기
python manage.py runserver
```
- 확인: `http://localhost:8000/api/`, 관리자 화면 `http://localhost:8000/admin/`
- 끄기: 실행 중인 터미널에서 `Ctrl + C`
- 계정이 없다면 `python manage.py createsuperuser`, 샘플 데이터는 `python manage.py seed_demo`
- `daphne`/`channels`가 `INSTALLED_APPS`에 등록되어 있어 `runserver` 명령 하나로 REST API와 실시간 채팅 WebSocket(`ws/chat/{room_id}/`)이 같은 프로세스에서 함께 떠 있다. 별도 프로세스를 더 켤 필요는 없다.
- 지갑 충전(토스페이먼츠 테스트 연동)을 쓰려면 `backend/.env`에 `TOSS_CLIENT_KEY`/`TOSS_SECRET_KEY`(developers.tosspayments.com 가입 후 API 키 메뉴에서 발급, 사업자등록 불필요)를 설정해야 한다. 비워두면 해당 기능만 동작하지 않고 나머지 기능(회원/상품/채팅/내부 잔액 송금 등)은 그대로 쓸 수 있다.

### ③ React 프론트엔드 (Vite)

```bash
# 프로젝트 루트에서
npm run dev
```
- 확인: `http://localhost:5173`
- 끄기: 실행 중인 터미널에서 `Ctrl + C`
- `vite.config.js`의 프록시 설정으로 `/api` 요청이 자동으로 `localhost:8000`(백엔드)으로 전달된다.

### 전체 켜기 순서 요약
| 순서 | 요소 | 명령 |
|---|---|---|
| 1 | PostgreSQL | `docker compose up -d postgres` |
| 2 | Django 백엔드 | `cd backend && source .venv/bin/activate && python manage.py runserver` |
| 3 | React 프론트엔드 | `npm run dev` (프로젝트 루트) |

### 전체 끄기 순서 요약
| 순서 | 요소 | 명령 |
|---|---|---|
| 1 | React 프론트엔드 | 터미널에서 `Ctrl + C` |
| 2 | Django 백엔드 | 터미널에서 `Ctrl + C` |
| 3 | PostgreSQL | `docker compose stop postgres` (또는 `docker compose down`) |

## 7. 구현 현황 (2026-07-21 기준)

1~5장은 "요구사항 도출 ~ 시스템 설계" 산출물이고, 이 장은 "시스템 구현" 단계에서 실제로 완성된 부분과 아직 설계로만 남아있는 부분을 구분해 기록한다. 코드가 바뀔 때마다 갱신하는 살아있는 문서이며, 아래 내용은 실제로 curl/브라우저로 동작을 확인한 것만 "완료"로 표시했다.

### 7.1 완료된 기능

**회원/계정**
- DRF Token 기반 회원가입/로그인/로그아웃(`POST /api/auth/signup`, `/login`, `/logout`). 프론트 `AuthContext`가 토큰을 `sessionStorage`에 저장해 새로고침해도 로그인이 유지된다 (탭/창마다 독립적이라 여러 계정으로 동시 테스트해도 토큰이 서로 덮어쓰이지 않는다 — 원래 `localStorage`였다가 변경).
- 비밀번호 변경(`POST /api/auth/password`, 현재 비밀번호 확인 후 변경 + 기존 토큰 폐기·새 토큰 발급)과 회원 탈퇴(`DELETE /api/users/me`)를 설정 › 계정 관리 화면에서 실제로 수행할 수 있다.
- 미구현: 소셜 로그인(OAuth — 프로바이더 앱 등록/시크릿 발급이 필요해 이번 범위 밖), 이메일 기반 "비밀번호 찾기"(발송 인프라 없음).

**상품 등록/조회/수정/삭제**
- `ProductCreate` 화면에서 실제 `<input type=file>`로 사진을 선택해 `POST /products` → `POST /products/{id}/images` 순서로 DB·디스크에 실제 저장한다.
- 같은 화면이 수정 모드(`/product/:id/edit`)도 겸한다 — 기존 사진 삭제, ‹›버튼으로 순서 변경(대표 사진 지정 포함), 제목/카테고리/상태/거래방식/설명 수정을 `PATCH /products/{id}`로 저장한다.
- 상품 상세에서 소유자에게만 수정/삭제 버튼과 판매중·예약중·판매완료 상태 전환 칩이 보인다(`DELETE /products/{id}`, `PATCH /products/{id}/status`).
- 메인 그리드는 `IntersectionObserver`로 스크롤 하단에 닿으면 다음 페이지(`?page=`)를 이어붙이는 무한 스크롤이 실제로 동작한다(백엔드 `PageNumberPagination`, 페이지당 12개).

**찜 / 신고**
- 찜 버튼이 `GET/POST/DELETE /api/wishlists`로 실제 동작하며 새로고침해도 상태가 유지된다.
- 설정 › 찜 목록에서도 찜한 상품 전체를 목록으로 조회하고 개별 해제할 수 있다 (`GET /api/wishlists`, `DELETE /api/wishlists/{id}`) — 이전에는 메뉴를 눌러도 메인화면으로만 이동했다.
- 신고 버튼이 사유를 입력받아 `POST /api/reports`로 실제 접수되고, 설정 › 신고 내역에서 본인이 신고한 내역과 처리 상태를 조회할 수 있다.
- **자동 차단/휴면 전환**: 같은 대상(상품 또는 유저)을 서로 다른 신고자 3명 이상이 신고하면 자동으로 제재된다 — 상품은 `is_blocked=True`로 표시되어 목록/상세에서 즉시 숨겨지고(관리자만 계속 조회 가능), 유저는 `is_active=False`로 전환되어 로그인이 막힌다. 관리자의 수동 신고 처리(처리완료 처리)와는 별개로 동작한다.

**친구 관리**
- 설정 › 친구 관리가 더 이상 mock 데이터를 쓰지 않는다. 아이디로 사용자 검색(`GET /users/lookup`) → 친구 요청(`POST /friends/requests`) → 받은/보낸 요청함에서 수락·거절·취소(`PATCH`/`DELETE /friends/requests/{id}`) → 수락된 친구 목록(`GET /friends/requests/accepted`)까지 전 과정이 실제 API로 동작한다.
- 친구 요청은 받은 사람만 수락/거절할 수 있도록 서버에서 검증한다.

**채팅 (실시간)**
- Django Channels + Daphne를 도입해 `python manage.py runserver` 한 프로세스가 REST API와 WebSocket을 동시에 서빙한다.
- 하단 네비 "채팅" 탭이 채팅방 목록 화면(`/chats`, `GET /api/chatrooms`)으로 연결되어 내가 참여 중인 모든 채팅방을 상대방/상품/거래 상태와 함께 보여준다 (이전에는 특정 상품의 채팅방으로 바로 진입했다).
- `POST /api/chatrooms`가 상품 기준으로 채팅방을 get-or-create한다.
- `ws/chat/{room_id}/?token=...`으로 접속해 텍스트 메시지가 실시간 양방향으로 오가고, 새로고침해도 히스토리가 유지된다.
- 채팅 내 사진 전송: `POST /chatrooms/{id}/images`로 업로드해 받은 URL을 WebSocket으로 `message_type=image`로 전송, 상대방 화면에 즉시 렌더링된다.
- 대리인(판매자 측·구매자 측 모두) 초대가 실제 참여자 목록에 반영되고, 참여가 시스템 메시지로 방 전체에 실시간 브로드캐스트된다. 거래 완료 후에도 대리인은 자동 퇴장하지 않고 채팅방에 남는다(분쟁 시 재확인 대비).
- 대면 거래 인증: 판매자 → (판매자 대리인) → (구매자 대리인) → 구매자 체인에서, 각 구간 양쪽이 각자 버튼(판매인증/인수인증/인계인증/구매인증)을 눌러야 다음 구간이 열리는 순차 로직이 있다.
  - 각 버튼 클릭은 `POST /chatrooms/{id}/verification`으로 `VerificationStep`에 기록되고, 새로고침해도 `GET /chatrooms/{id}/verification`으로 진행 상태가 복원된다. 구간 양쪽이 다 채워지면 `TradeVerification` 레코드가 생성되고 채팅방 상태(`negotiating→handover_done/confirmed→done`)가 서버에 반영되며, WebSocket으로 다른 참여자 화면에도 실시간 전파된다 — 더 이상 브라우저 로컬 상태에만 머무르지 않는다.
  - **판매자 대리인 초대 지원**: 대리인이 판매자 측인지 구매자 측인지는 클라이언트가 지정하지 않고, `POST /chatrooms/{id}/participants`를 호출한 사람 본인이 채팅방에서 판매자인지 구매자인지를 서버가 보고 자동으로 `seller_agent`/`buyer_agent` 역할을 부여한다(대리인 본인은 또 다른 대리인을 초대할 수 없고, 같은 쪽 대리인 슬롯은 1명으로 제한). 초대 순서와 무관하게 인증 체인은 항상 판매자→판매자 대리인→구매자 대리인→구매자 순으로 정렬된다.

**알림 설정**
- 설정 › 알림 설정의 채팅/거래 상태 토글이 `GET/PATCH /notifications/settings`로 서버에 저장되고 새로고침해도 유지된다. 다만 실제 푸시 발송(FCM/APNs)은 하지 않는다 — "받을지 여부"만 저장하는 수준.

**관리자**
- `/admin` 화면(프론트, `is_staff` 사용자에게만 노출)에서 통계, 전체 사용자 목록(활성/정지 토글), 전체 상품 목록(강제 삭제), 전체 신고 목록(처리완료 처리)을 다룬다.
- 전용 REST API(`/admin/users`, `/admin/products`, `/admin/reports`, `/admin/stats`)는 `IsStaffUser` 권한으로 보호되며, Django 기본 admin 사이트(`/admin/`, HTML)와는 별개다.

**송금**
- 가입 시 프로필에 잔액(`balance`, 기본 1,000,000)이 지급되고, 설정 › 송금 화면 또는 채팅방(구매자→판매자) 안의 "송금하기" 버튼으로 `POST /transfers`를 호출해 실제로 잔액이 이동한다.
- 설정 › 송금 화면에서 내 잔액과 보내거나 받은 전체 내역(상대방, 금액, 메모)을 조회할 수 있다.
- 채팅방에서 보낸 송금은 `chatroom`으로 거래 맥락에 연결되고, 완료 시 시스템 메시지로 방에 브로드캐스트된다.
- 자기 자신에게 송금, 0원 이하 송금, 잔액 초과 송금은 모두 서버에서 거부된다. 동시에 여러 송금 요청이 들어와도 잔액이 음수가 되지 않도록 DB 트랜잭션과 행 잠금으로 처리한다 (아래 7.2절 참고).
- **지갑 충전(토스페이먼츠 테스트 연동)**: 설정 › 송금 › "충전하기" 버튼으로 `/wallet/topup` 화면에 진입해 실제 결제창(테스트 모드)을 열고, 토스 문서의 테스트 카드번호로 결제를 완료하면 `/wallet/topup/result`에서 서버가 결제 승인을 확인한 뒤 잔액을 충전한다. 실제 결제는 발생하지 않는다.

### 7.2 설계 대비 변경/추가된 아키텍처

- **실시간 통신**: 4장의 데이터 흐름도에 있던 WebSocket이 실제로 Django Channels로 구현됐다. 다만 개발 환경은 `channels.layers.InMemoryChannelLayer`를 쓰므로 **단일 프로세스에서만 동작**한다 — 워커를 여러 개 띄우거나 서버를 늘리려면 Redis 기반 채널 레이어(`channels_redis`)로 교체해야 한다.
- **인증 방식**: 세션 쿠키 대신 DRF `TokenAuthentication`을 채택했다. 브라우저 WebSocket API가 커스텀 헤더를 못 보내는 제약 때문에, 토큰을 쿼리스트링(`?token=`)으로 전달받아 인증하는 미들웨어(`marketplace/ws_auth.py`)를 별도로 구현했다.
- **DB**: `rest_framework.authtoken` 앱을 통해 `authtoken_token` 테이블이 추가됐다 (3장의 DB 설계에는 없던 테이블, 발급된 로그인 토큰을 저장). `verification_steps`, `notification_settings` 테이블도 구현 과정에서 추가됐다 (3장에 반영 완료).
- **권한 보안 수정**: 상품 이미지 업로드(`POST /products/{id}/images`)와 상태 변경(`PATCH /products/{id}/status`)에 소유자 확인이 빠져 있어 로그인만 하면 남의 상품도 건드릴 수 있던 취약점을 발견해 `IsOwnerOrReadOnly`를 추가했다.
- **대면 거래 인증 위조 방지**: `POST /chatrooms/{id}/verification`이 `from_user`/`to_user`를 클라이언트 입력값으로 그대로 신뢰하던 것을 서버가 참여자 목록에서 직접 계산한 체인(판매자→대리인→구매자 순, `verification_chain`)으로 바꿔, `hop_index`/`side`로 결정되는 "지금 눌러야 할 사람"과 실제 요청자(`request.user`)가 다르면 403으로 거부하도록 수정했다 — 구매자가 판매인증 버튼을 대신 누르는 식의 위조를 서버에서 막는다.
- **사진 순서 변경의 동시성 이슈**: `product_images`에 `(product_id, order)` 유니크 제약이 있어, 두 사진의 순서를 서버에서 그대로 맞바꾸면 중간에 값이 겹쳐 `IntegrityError`가 발생했다. `PATCH /products/{id}/images/order`는 영향받는 사진들을 먼저 충돌 없는 임시 값(1만 이상)으로 옮긴 뒤 최종 순서를 저장하는 2단계 갱신으로 처리한다.
- **가격/송금 요구사항 범위 조정**: `products` 테이블에는 애초에 가격(price) 컬럼이 없었고, 실제 결제 PG 연동 없이는 "사용자 간 송금"을 의미 있게 구현하기 어려웠다. `products.price`, `profiles.balance`, `transfers` 테이블을 추가해 **내부 포인트/잔액 기반 송금**으로 요구사항을 좁혀 구현했다(에스크로/보류결제는 여전히 범위 밖, 아래 7.3 참고). 송금은 두 `Profile.balance` 행을 `user_id` 오름차순으로 `select_for_update`한 뒤 같은 트랜잭션에서 검증→차감→입금을 수행해, 두 사람이 동시에 서로에게 송금해도 잠그는 순서가 엇갈려 교착상태(deadlock)에 빠지거나 잔액이 음수가 되는 일이 없도록 했다.
- **신고 자동 차단 로직 누락 발견 및 추가**: 1장 요구사항과 25p 시스템 설계에 "일정 횟수 이상 신고된 상품/유저는 자동 차단/휴면 전환"이 명시돼 있었는데, 실제로는 신고 접수와 관리자 수동 처리만 구현돼 있고 자동화 로직 자체가 빠져 있었다. `products.is_blocked` 필드를 추가하고, 임계값 기준을 "신고 횟수"가 아니라 **"서로 다른 신고자 수"(3명 이상)**로 설계해 한 명이 같은 대상을 반복 신고해서 혼자 임계값을 채우는 어뷰징을 막았다(`views.py`의 `apply_auto_moderation`).
- **PG 연동 방식 정정(설계 중 발견)**: 처음엔 토스페이먼츠 "결제위젯"(`widgets()`/`renderPaymentMethods()`, `gck`/`gsk` 키) 방식으로 설계했으나, 실제 발급받은 키가 `test_ck_...`(API 개별연동/결제창 키)여서 그대로는 `INVALID_API_KEY` 오류가 났을 것이다. `GET /payments/config`로 키 프리픽스를 먼저 확인한 뒤 `tossPayments.payment({customerKey}).requestPayment({method:'CARD', ...})` 방식(API 개별연동)으로 다시 구현했다 — 키 종류와 SDK 호출 방식이 반드시 짝이 맞아야 한다.

### 7.3 아직 설계로만 남아있는 영역

| 영역 | 상태 |
|---|---|
| 소셜 로그인(OAuth) | 미구현 — 카카오/구글 등 프로바이더에 앱을 등록하고 client id/secret을 발급받아야 진행 가능 |
| 에스크로(안전결제) | 미구현 — 송금 자체(내부 잔액 이체)는 구현했지만, "거래 확인 전까지 금액을 보류했다가 확정 시 릴리즈"하는 에스크로 로직은 별도 상태 머신이 필요해 이번 범위에서는 보류. 대면 거래 인증(7.1절)과 별개로 동작한다 |
| 푸시 알림 발송 | 알림 "설정 저장"까지는 구현했지만, FCM/APNs로 실제 기기에 푸시를 보내는 부분은 미구현 (Firebase/Apple 자격증명 필요) |

### 7.4 실서비스 전환 시 필요한 인프라 (요약)

| 구성요소 | 지금(개발) | 필요한 것 |
|---|---|---|
| DB | 로컬 Docker Postgres | 관리형 PostgreSQL (Supabase/Neon/RDS 등) |
| 채널 레이어 | `InMemoryChannelLayer` | Redis (`channels_redis`) — 워커가 2개 이상이면 필수 |
| 이미지 저장 | 로컬 디스크(`backend/media/`) | 오브젝트 스토리지(S3 등) + `django-storages` |
| 백엔드 호스팅 | `manage.py runserver` | WebSocket 상시 연결을 지원하는 호스트 (Render/Railway/Fly.io/VM) |
| 프론트 호스팅 | Vite dev 서버 | 정적 빌드 후 Vercel/Netlify/Cloudflare Pages |
| 환경설정 | `DEBUG=True`, 로컬 `ALLOWED_HOSTS` | `DEBUG=False`, 실제 도메인, HTTPS(WSS) |
